## Сценарии пользователя и тестирование

### Пользовательская история 1 — Ввод данных в табличном реестре

Пользователь видит таблицу, где строки — показатели здоровья, столбцы — замеры по датам. Для добавления нового замера нажимает «Новый замер» — появляется столбец слева с полем даты (по умолчанию — сегодня) и пустыми ячейками для показателей. Часть показателей (например, рост) заполняется из предыдущего замера. Ячейки редактируются inline — клик по ячейке открывает ввод. При вводе значения в ячейке справочно показывается предыдущее значение и отклонение в абсолютных единицах и процентах.

**Почему этот приоритет**: Без ввода данных невозможно пользоваться функцией — это основа всего модуля.

**Независимое тестирование**: Можно полностью протестировать, добавив новый замер в таблицу и заполнив несколько ячеек. Ценность — пользователь уже может вести журнал здоровья и видеть всю историю.

**Сценарии приёма**:

1. **Дано** пользователь на экране таблицы замеров, **Когда** он нажимает «Новый замер», заполняет дату и ячейки показателей, **Тогда** данные сохраняются, появляется новый столбец в таблице
2. **Дано** пользователь ранее вводил данные, **Когда** он добавляет новый замер, **Тогда** показатели с признаком автозаполнения заполнены из предыдущей записи, остальные ячейки пустые
3. **Дано** пользователь вводит значение в ячейку, **Когда** ячейка в фокусе ввода, **Тогда** рядом с ячейкой или во всплывающей подсказке показывается предыдущее значение этого показателя и отклонение в абсолютных единицах и процентах (например: «72 кг → 73 кг (+1 кг, +1.4%)»)

---

### Пользовательская история 2 — Редактирование и удаление данных прямо в таблице

Пользователь может исправить любое значение прямо в ячейке таблицы (включая очистку — установка пустого значения), а также удалить целый замер (столбец).

**Почему этот приоритет**: Ошибки ввода неизбежны — возможность исправления критична для доверия к данным.

**Независимое тестирование**: Кликнуть ячейку → изменить значение → проверить, что обновилось. Удалить столбец → убедиться, что исчез.

**Сценарии приёма**:

1. **Дано** пользователь видит ячейку с некорректным значением, **Когда** он кликает по ячейке, вводит новое значение и подтверждает (Enter или клик вне ячейки), **Тогда** значение обновлено в таблице и в хранилище
2. **Дано** пользователь на экране таблицы, **Когда** он выбирает удаление замера (столбца) и подтверждает, **Тогда** столбец удалён из таблицы

---

### Пользовательская история 3 — Просмотр динамики показателей на графиках

Пользователь переключается в режим «Графики», выбирает набор показателей и видит графики их изменения во времени. По оси X — время, по оси Y — значения. Каждый показатель на своей шкале.

**Почему этот приоритет**: Визуализация — ключевая ценность модуля, ради которой пользователь ведёт журнал.

**Независимое тестирование**: Внести 3+ замеров → переключиться в режим графиков → выбрать 2–3 показателя → убедиться, что графики отображаются корректно с разными шкалами.

**Сценарии приёма**:

1. **Дано** пользователь имеет минимум 2 замера, **Когда** он переключается в режим «Графики» и выбирает показатели, **Тогда** отображаются линейные графики с общей осью X (время) и индивидуальными шкалами Y для каждого показателя, по умолчанию — за весь период
2. **Дано** пользователь в режиме графиков, **Когда** он перемещает ползунки начала/конца периода, **Тогда** график перерисовывается, показывая только замеры в выбранном диапазоне
3. **Дано** пользователь в режиме графиков, **Когда** он добавляет или убирает показатель из отображения, **Тогда** график мгновенно перерисовывается
4. **Дано** у пользователя только 1 замер в выбранном периоде, **Когда** он переключается в режим графиков, **Тогда** отображается сообщение «Для построения графика нужно минимум 2 замера»

---

### Пользовательская история 4 — Расширение набора показателей

Разработчик может добавить новый показатель без переписывания основной логики. Новый показатель появляется в таблице (новая строка), в форме ввода и на графиках.

**Почему этот приоритет**: Низкий — работает на масштабируемость, не блокирует основную функциональность.

**Независимое тестирование**: Добавить новый показатель в конфигурацию → убедиться, что он появился как новая строка в таблице и доступен на графиках.

**Сценарии приёма**:

1. **Дано** добавлен новый показатель в конфигурацию, **Когда** пользователь открывает таблицу замеров, **Тогда** новый показатель отображается как новая строка
2. **Дано** сохранены замеры с новым показателем, **Когда** пользователь открывает графики, **Тогда** новый показатель доступен для выбора и отображения

### Граничные случаи

- Что произойдёт, если пользователь вводит некорректные значения (отрицательный вес, пульс 0, давление 300/200)? — Система должна показывать предупреждение прямо в ячейке при выходе за разумные границы
- Как система обрабатывает некорректный формат давления (например, «120/» или «/80»)? — Поле подсвечивается как ошибка, значение не сохраняется
- Как система обрабатывает некорректный формат сна (например, «6ч30м», «6:30»)? — Система распознаёт основные форматы и нормализует, при невозможности — показывает подсказку с ожидаемым форматом
- Что произойдёт, если пользователь вводит данные за дату, на которую уже есть замер? — Существующий замер полностью заменяется новым
- Как система обрабатывает ситуацию, когда нет предыдущего замера для автозаполнения? — Ячейки с признаком автозаполнения остаются пустыми, отклонения не показываются
- Что если пользователь удаляет единственный замер? — Замер удаляется, таблица пуста
- Минимальное количество замеров для графиков — 2, максимум — все доступные
- Что если таблица не помещается по ширине при большом количестве замеров? — Горизонтальная прокрутка с фиксацией первого столбца (показатели)
- Что если в выбранном периоде нет замеров? — Показывать сообщение «Нет данных за выбранный период»

## Экранные формы

### Форма 1: Таблица-реестр замеров и графики (единый экран)

**Статус**: НОВАЯ
**Связанная история**: Пользовательские истории 1, 2 и 3

**Описание**: Основной экран модуля. Переключатель режимов «Таблица» / «Графики» в верхней части. По умолчанию — режим таблицы.

**Режим «Таблица»**: таблица, где строки — показатели здоровья, столбцы — замеры по датам. Первый столбец (показатели) зафиксирован при горизонтальной прокрутке.

**Режим «Графики»**: линейные графики изменения показателей во времени с общей осью X (время) и индивидуальными шкалами Y для каждого показателя.

**Элементы управления**:

| Элемент | Тип | Обязательный | Описание |
|---------|-----|:-----------:|----------|
| Переключатель вида | Табы / сегментированный переключатель | — | Переключение между режимами «Таблица» и «Графики» |
| Ячейка таблицы | Редактируемое поле | Нет | Значение показателя за определённую дату. Клик → режим редактирования (в режиме таблицы) |
| Заголовок столбца | Дата + меню | — | Дата замера. Контекстное меню: «Удалить замер» (в режиме таблицы) |
| Заголовок строки | Текст | — | Название показателя с единицей измерения (например, «Вес, кг») (в режиме таблицы) |
| Кнопка «Новый замер» | Кнопка | — | Добавить столбец нового замера слева (в режиме таблицы) |
| Выбор показателей | Чекбоксы / переключатели | Да | Множественный выбор показателей для отображения на графиках (в режиме графиков) |
| Фильтр периода | Два ползунка на оси времени | Нет | Временной интервал для отображения на графиках. Отображается ось от первого до последнего замера, на ней два перемещаемых ползунка — начало и конец периода. По умолчанию — за весь период |
| Область графика | Графический элемент | — | Линейные графики с общей осью X (время) и индивидуальными шкалами Y (в режиме графиков) |

**Логика работы**:

**Режим «Таблица»**: таблица отображает все сохранённые замеры. Столбцы отсортированы по дате — новые слева. Показатели отображаются в фиксированном порядке, определённом в справочнике, с разделительными линиями между логическими блоками (антропометрия, кардио, активность, сон, нагрузочные тесты). Первый столбец (показатели) зафиксирован при горизонтальной прокрутке. При нажатии «Новый замер» добавляется пустой столбец слева: поле даты (по умолчанию — сегодня) и пустые ячейки показателей. Показатели, для которых в справочнике установлен признак автозаполнения, заполняются значениями из предыдущего замера с возможностью изменения. Ячейки показателей разных типов: простые числовые (вес, пульс, шаги, ккал) — ввод числа; составные (давление) — ввод двух чисел через слэш в одном поле; длительность (сон) — ввод в формате «X ч Y м». Пользователь кликает ячейку — она переходит в режим редактирования. При вводе значения показывается подсказка с предыдущим значением и отклонением (абсолютным и процентным). Валидация происходит inline — при выходе за разумные границы ячейка подсвечивается. Изменение сохраняется при потере фокуса или нажатии Enter. Очистка ячейки (пустое значение) — допустимое действие. Удаление столбца — через контекстное меню заголовка с подтверждением. При переключении режима или действиях, ведущих к потере несохранённых изменений, — запрос подтверждения.

**Режим «Графики»**: при переключении отображаются линейные графики. Если есть несохранённые изменения в таблице, запрашивается подтверждение. По умолчанию показываются данные за весь доступный период. Под графиками — временная шкала от первого до последнего замера с двумя ползунками (начало и конец периода). Пользователь перемещает ползунки — график мгновенно перерисовывается, показывая только замеры в выбранном диапазоне. Все показатели выбраны по умолчанию (если есть данные). Пользователь может включать/отключать отдельные показатели через чекбоксы — график перерисовывается мгновенно. По оси X — дата замера, по оси Y — значения. У каждого показателя своя шкала Y (подписана единицами измерения). Если замеров меньше 2 в выбранном периоде — показывается информационное сообщение «Для построения графика нужно минимум 2 замера». Выбранные показатели и период запоминаются до следующего переключения.

## Требования

### Функциональные требования

- **FR-001**: Система ДОЛЖНА отображать таблицу замеров, где строки — показатели, столбцы — замеры по датам
- **FR-002**: Система ДОЛЖНА позволять добавлять новые замеры как столбцы в таблицу
- **FR-003**: Система ДОЛЖНА автоматически заполнять из предыдущего замера те показатели, для которых в справочнике установлен признак автозаполнения
- **FR-004**: Система ДОЛЖНА показывать предыдущие значения показателей и отклонение в абсолютных единицах и процентах при вводе данных
- **FR-005**: Система ДОЛЖНА поддерживать inline-редактирование ячеек таблицы
- **FR-006**: Система ДОЛЖНА валидировать значения и показывать предупреждение при выходе за разумные границы
- **FR-007**: Система ДОЛЖНА сохранять все внесённые замеры локально в JSON-файлах
- **FR-008**: Пользователь ДОЛЖЕН иметь возможность удалять любой замер целиком
- **FR-009**: При вводе данных за дату с существующим замером система ДОЛЖНА заменить старый замер новым целиком
- **FR-010**: Система ДОЛЖНА отображать графики изменения показателей во времени с индивидуальными шкалами для каждого показателя
- **FR-011**: Пользователь ДОЛЖЕН иметь возможность выбирать, какие показатели отображать на графиках
- **FR-012**: Пользователь ДОЛЖЕН иметь возможность задавать временной интервал для отображения графиков; по умолчанию — за весь период
- **FR-013**: Система ДОЛЖНА поддерживать расширение набора показателей путём добавления конфигурации разработчиком
- **FR-014**: Первый столбец таблицы (показатели) ДОЛЖЕН быть зафиксирован при горизонтальной прокрутке
- **FR-015**: Артериальное давление ДОЛЖНО вводиться в одном поле в формате «XXX/XX» (два числа через слэш)
- **FR-016**: Продолжительность сна ДОЛЖНА вводиться в формате «X ч Y м» (например, «6 ч 30 м») и храниться в минутах
- **FR-017**: При переключении режима или действиях, ведущих к потере несохранённых изменений, система ДОЛЖНА запросить подтверждение пользователя

### Ключевые сущности

- **HealthEntry (Замер здоровья)**: Единица данных за одну дату. Содержит дату и набор показателей. Каждый показатель хранится в формате, удобном для ввода и отображения:
  - Простые числовые значения: рост (см), вес (кг), пульс (уд/мин), шаги, ккал, этажи, отжимания
  - Составные значения: артериальное давление — два целых числа (систолическое/диастолическое), хранится как пара
  - Длительность: сон — целое число минут (вводится как «X ч Y м», отображается в том же формате)
  - Показатели с признаком автозаполнения (например, рост) копируются из предыдущего замера при создании нового.
  - Одна запись на дату.

- **MetricDefinition (Определение показателя)**: Метаданные показателя — название, единица измерения, тип ввода (простое число / составное значение / длительность), разумные границы значений, признак автозаполнения из предыдущего замера, порядок отображения в таблице. Используется для расширения набора показателей.

## Допущения

- Все данные хранятся локально в JSON-файлах (аналогично profile.json), без сетевой синхронизации
- Один пользователь — один журнал здоровья (привязан к профилю)
- Валидация «разумных границ» использует стандартные медицинские диапазоны (например, вес 30–300 кг, пульс 30–200 уд/мин)
- При вводе данных за дату, на которую уже есть замер, система заменяет существующие данные целиком. Предыдущий замер полностью удаляется и заменяется новым.
- Для визуализации графиков используется стандартная JS-библиотека графиков (например, Chart.js или Recharts) — выбор будет определён на этапе планирования
- Расширение набора показателей (FR-012) выполняется разработчиком через добавление записи в конфигурационный массив в коде — пользователь не управляет набором показателей самостоятельно.
- Данные за предыдущий замер определяются как самая свежая запись по дате до выбранной
- При большом количестве замеров таблица поддерживает горизонтальную прокрутку с фиксацией первого столбца

# Задачи: Модуль «Планы»

**Входные данные**: Проектная документация из `/specs/006-future-plans/`
**Предварительные требования**: plan.md, spec.md, data-model.md, contracts/, research.md, quickstart.md

**Тесты**: Включены unit-тесты для критической бизнес-логики (фильтрация, сортировка, определение статуса «просрочено», переходы состояний) согласно конституции проекта.

**Организация**: Задачи сгруппированы по пользовательским историям, чтобы обеспечить независимую реализацию и тестирование каждой истории.

## Формат: `[ID] [P?] [Story] Описание`

- **[P]**: Можно запускать параллельно (разные файлы, нет зависимостей)
- **[Story]**: Какой пользовательской истории принадлежит задача (например, US1, US2, US3)
- Включайте точные пути к файлам в описаниях

## Соглашения о путях

Один проект (Tauri + React):
- Frontend: `src/`
- Backend (Rust/Tauri): `src-tauri/src/`
- Типы: `src/types/`
- Тесты: unit-тесты рядом с исходными файлами (`__tests__/`)

## Зависимости и порядок выполнения

### Зависимости фаз

- **Настройка (Фаза 1)**: Нет зависимостей — можно начинать немедленно
- **Фундаментальные (Фаза 2)**: Зависит от завершения Настройки — **БЛОКИРУЕТ** все пользовательские истории
- **Пользовательские истории (Фаза 3+)**: Все зависят от завершения Фундаментальной фазы
  - US1 (P1) → US2 (P2) → US3 (P3) → US4 (P4) — последовательная рекомендуемая выдача
  - После Фазы 2 все US могут разрабатываться параллельно
- **Полировка (Фаза 7)**: Зависит от завершения всех пользовательских историй

### Зависимости пользовательских историй

| История | Зависит от | Может начаться после |
|---------|-----------|---------------------|
| US1 — Реестр | Фаза 2 | T003-T009 завершены |
| US2 — Создание | Фаза 2 + US1 | US1 готова (нужен реестр для отображения) |
| US3 — Выполнение/Отмена | Фаза 2 + US1 | US1 готова (нужна задача для действия) |
| US4 — Детальная карточка | Фаза 2 + US1 | US1 готова (нужен реестр для клика) |

### Внутри каждой пользовательской истории

- Модели/типы до сервисов
- Сервисы до компонентов UI
- Базовые компоненты до интегрированных страниц
- История завершена до перехода к следующему приоритету

### Возможности для параллелизма

- **Фаза 1**: T001-T002 — все [P], можно параллельно

## Стратегия реализации

### Сначала MVP (US1 — Реестр)

1. Завершить Фазу 1: Настройка
2. Завершить Фазу 2: Фундаментальные (КРИТИЧНО — блокирует все истории)
3. Завершить Фазу 3: Пользовательская история 1
4. **ОСТАНОВИТЬСЯ и ПРОВЕРИТЬ**: Реестр загружается, сортирует, фильтрует
5. Развернуть/продемонстрировать

### Инкрементальная выдача

1. Фаза 1 + Фаза 2 → CRUD и утилиты готовы
2. + US1 → Реестр работает → MVP!
3. + US2 → Создание задач → Демо
4. + US3 → Выполнение/Отмена → Демо
5. + US4 → Детальная карточка с редактированием → Демо
6. Фаза 7 → Полировка, тесты, проверка quickstart

### Стратегия параллельной команды

С несколькими разработчиками:

1. Команда завершает Настройку + Фундаментальные вместе
2. После завершения Фундаментальных:
   - Разработчик A: US1 (Реестр) → US4 (Карточка)
   - Разработчик B: US2 (Создание) + US3 (Выполнение/Отмена)
3. Истории завершаются и интегрируются независимо

# Спецификация функции: Ведение информации о приёмах врача

**Ветка функции**: `003-doctor-visits`
**Создана**: 9 апреля 2026
**Статус**: Черновик
**Входные данные**: Описание пользователя: "Ведение информации о приёмах врача (фактические приёмы). Состав информации: Дата, ФИО врача, специальность, клиника, основные результаты (текстом), назначенные препараты (текстом), назначенные другие процедуры и исследования (текстом), скан заключения. Варианты ввода – либо вручную, либо путем отправки скана приёма в LLM для распознавания и автоматического заполнения полей (с возможностью изменения вручную). Оценка врача и качества приёма – по 5-балльной шкале. Отдельный раздел в меню с реестром приёмов (плиточный интерфейс), добавление/изменение записей о приемах в модальном окне, удаление записей. Поиск по периоду, специальности врача. По умолчанию все, последние вверху. Карточка приёма с более полной информацией. Отображение ссылки на предыдущий и следующий приём (по ФИО и специальность врача)."

## Сценарии пользователя и тестирование *(обязательно)*

### Пользовательская история 1 — Ручное добавление записи о приёме

Пользователь может вручную заполнить форму с информацией о приёме врача: дата, ФИО врача, специальность, клиника, результаты приёма, назначенные препараты, другие процедуры и исследования, скан заключения. После сохранения запись появляется в реестре приёмов. Пользователь также может оценить врача и качество приёма по 5-балльной шкале.

**Почему этот приоритет**: Это основной сценарий — без возможности добавления записей вся остальная функциональность бессмысленна. MVP с одним лишь ручным вводом уже приносит ценность.

**Независимое тестирование**: Можно полностью протестировать, заполнив форму вручную, сохранив запись и убедившись, что она корректно отображается в реестре и карточке приёма.

**Сценарии приемки**:

1. **Дано** пользователь находится на странице реестра приёмов, **Когда** пользователь нажимает «Добавить приём», заполняет обязательные поля (дата, ФИО врача, специальность) и нажимает «Сохранить», **Тогда** запись появляется в реестре первой (самой новой)
2. **Дано** пользователь открыл форму добавления приёма, **Когда** пользователь не заполнил обязательное поле и нажал «Сохранить», **Тогда** система отображает сообщение об ошибке с указанием незаполненного поля
3. **Дано** пользователь открыл форму добавления приёма, **Когда** пользователь заполняет все поля, включая оценку врача (1–5), и нажимает «Сохранить», **Тогда** запись сохраняется с указанной оценкой

---

### Пользовательская история 2 — Реестр приёмов с поиском и фильтрацией

Пользователь видит отдельный раздел меню «Приёмы» с плиточным отображением всех записей о приёмах. По умолчанию отображаются все приёмы, отсортированные по дате (новые сверху). Пользователь может фильтровать по периоду дат и специальности врача.

**Почему этот приоритет**: Реестр — основной способ навигации по записям; без него пользователь не может находить и просматривать свои приёмы.

**Независимое тестирование**: Можно протестировать, создав несколько записей и проверив, что они корректно отображаются плитками, сортируются по дате и фильтруются по периоду и специальности.

**Сценарии приемки**:

1. **Дано** в системе есть 3 записи о приёмах с разными датами, **Когда** пользователь открывает раздел «Приёмы», **Тогда** записи отображаются плитками, самая новая — сверху
2. **Дано** пользователь находится в реестре приёмов, **Когда** пользователь задаёт фильтр по периоду (дата начала — дата окончания), **Тогда** отображаются только приёмы в указанном диапазоне
3. **Дано** пользователь находится в реестре приёмов, **Когда** пользователь выбирает специальность врача из фильтра, **Тогда** отображаются только приёмы врачей выбранной специальности
4. **Дано** фильтры не применены, **Когда** пользователь нажимает «Сбросить», **Тогда** отображаются все приёмы

---

### Пользовательская история 3 — Автоматическое распознавание скана через LLM

Пользователь может загрузить скан (фотографию) заключения врача. Система отправляет изображение в LLM, которая распознаёт и автоматически заполняет поля формы: ФИО врача, специальность, клиника, результаты, препараты, процедуры. Пользователь может проверить и отредактировать распознанные данные перед сохранением.

**Почему этот приоритет**: Значительно ускоряет ввод данных и снижает вероятность ошибок при ручном вводе, но не блокирует MVP — ручной ввод работает без LLM.

**Независимое тестирование**: Можно протестировать, загрузив скан, проверив, что поля формы заполнены автоматически, и убедившись, что пользователь может отредактировать значения перед сохранением.

**Сценарии приемки**:

1. **Дано** пользователь открыл форму добавления приёма, **Когда** пользователь загружает изображение скана заключения, **Тогда** система отображает индикатор обработки и затем заполняет распознанные поля
2. **Дано** LLM вернула распознанные данные, **Когда** пользователь просматривает форму, **Тогда** все распознанные поля доступны для ручного редактирования
3. **Дано** LLM не смогла распознать часть данных, **Когда** обработка завершена, **Тогда** нераспознанные поля остаются пустыми, пользователь может заполнить их вручную

---

### Пользовательская история 4 — Просмотр карточки приёма и навигация

Пользователь может открыть карточку приёма с полной информацией. В карточке отображаются ссылки на предыдущий и следующий приём того же врача (по ФИО и специальности).

**Почему этот приоритет**: Позволяет отслеживать историю наблюдений у конкретного врача — важная часть пользовательского опыта.

**Независимое тестирование**: Можно протестировать, открыв карточку приёма и проверив наличие и корректность ссылок на соседние приёмы того же врача.

**Сценарии приемки**:

1. **Дано** пользователь находится в реестре приёмов, **Когда** пользователь кликает на плитку приёма, **Тогда** открывается карточка с полной информацией о приёме
2. **Дано** у врача (то же ФИО и специальность) есть несколько приёмов, **Когда** пользователь просматривает карточку приёма, **Тогда** отображаются ссылки на предыдущий и следующий приём этого врача
3. **Дано** у врача только один приём, **Когда** пользователь просматривает карточку, **Тогда** ссылки на предыдущий/следующий приём не отображаются

---

### Пользовательская история 5 — Редактирование и удаление записей

Пользователь может редактировать существующие записи о приёмах и удалять их. Редактирование открывается в том же модальном окне, что и создание. При удалении запрашивается подтверждение.

**Почему этот приоритет**: Необходимая функция управления данными, но не блокирует первичное использование.

**Независимое тестирование**: Можно протестировать, отредактировав запись и проверив обновление в реестре, а также удалив запись с подтверждением и проверив её исчезновение.

**Сценарии приемки**:

1. **Дано** пользователь находится в реестре приёмов, **Когда** пользователь выбирает запись и нажимает «Редактировать», **Тогда** модальное окно открывается с заполненными данными
2. **Дано** пользователь внёс изменения и нажал «Сохранить», **Тогда** обновлённая запись отображается в реестре
3. **Дано** пользователь нажимает «Удалить» на записи, **Когда** система запрашивает подтверждение и пользователь подтверждает, **Тогда** запись удаляется из реестра

---

### Граничные случаи

- Что произойдёт, если LLM вернёт некорректные или бессмысленные данные? Система должна позволить пользователю полностью очистить и перезаполнить поля.
- Как система обрабатывает загрузку повреждённого или неподдерживаемого файла изображения? Пользователь должен получить понятное сообщение об ошибке.
- Что происходит при попытке сохранить запись без обязательных полей (дата, ФИО врача, специальность)? Система блокирует сохранение и подсвечивает ошибки.
- Как отображаются плитки, если приём не содержит скана заключения? Плитка отображается с доступной текстовой информацией, поле скана — пустое.
- Что происходит, если при отправке скана в LLM возникает сетевая ошибка? Пользователь получает сообщение об ошибке и может повторить попытку или перейти к ручному вводу.

## Экранные формы (если применимо)

### Форма 1: Реестр приёмов

**Статус**: НОВАЯ
**Связанная история**: Пользовательская история 2

**Элементы управления**:

| Элемент | Тип | Обязательный | Описание |
|---------|-----|:-----------:|----------|
| Плитки приёмов | Контейнер-список (плиточный вид) | Да | Отображает карточки приёмов: дата, ФИО врача, специальность, клиника, оценка |
| Фильтр «Период» | Выбор диапазона дат | Нет | Позволяет задать дату начала и дату окончания для фильтрации |
| Фильтр «Специальность» | Выпадающий список | Нет | Список специальностей врачей из существующих записей |
| Кнопка «Сбросить фильтры» | Кнопка | Нет | Сбрасывает все применённые фильтры |
| Кнопка «Добавить приём» | Кнопка | Да | Открывает модальное окно создания записи |

**Логика работы**:

При открытии формы загружаются все записи, отсортированные по дате приёма (убывание — новые сверху). Плитки отображают краткую информацию: дата, ФИО врача, специальность, клиника, оценка (звёзды). Пользователь может применить фильтр по периоду и/или специальности — реестр обновляется. Кнопка «Сбросить фильтры» возвращает полный список. Клик по плитке открывает карточку приёма (Форма 3). Кнопка «Добавить приём» открывает модальное окно создания записи (Форма 2).

---

### Форма 2: Модальное окно создания/редактирования приёма

**Статус**: НОВАЯ
**Связанная история**: Пользовательская история 1, Пользовательская история 3, Пользовательская история 5

**Элементы управления**:

| Элемент | Тип | Обязательный | Описание |
|---------|-----|:-----------:|----------|
| Дата приёма | Поле выбора даты | Да | Дата приёма |
| ФИО врача | Текстовое поле | Да | Полное имя врача |
| Специальность | Текстовое поле | Да | Специальность врача (терапевт, кардиолог и т.д.) |
| Клиника | Текстовое поле | Нет | Название медицинского учреждения |
| Основные результаты | Многострочное текстовое поле | Нет | Текстовое описание результатов приёма |
| Назначенные препараты | Многострочное текстовое поле | Нет | Список назначенных лекарственных средств |
| Другие процедуры и исследования | Многострочное текстовое поле | Нет | Назначенные обследования, процедуры |
| Скан заключения | Загрузка файла (изображение) | Нет | Изображение/скан документа |
| Кнопка «Распознать скан» | Кнопка | Нет | Запускает LLM-распознавание загруженного скана |
| Оценка врача | Рейтинг (5 звёзд) | Нет | Оценка врача и качества приёма от 1 до 5 |
| Кнопка «Сохранить» | Кнопка | Да | Сохраняет запись (создание или обновление) |
| Кнопка «Отмена» | Кнопка | Да | Закрывает модальное окно без сохранения |
| Индикатор обработки | Спиннер/прогресс | Нет | Отображается во время LLM-распознавания |

**Логика работы**:

При открытии для создания все поля пусты. При открытии для редактирования поля заполнены данными существующей записи. Пользователь может ввести данные вручную или загрузить скан и нажать «Распознать скан» — после обработки LLM поля заполняются автоматически. Пользователь может изменить любое распознанное значение. При нажатии «Сохранить» выполняется валидация обязательных полей (дата, ФИО врача, специальность). При ошибках валидации поля подсвечиваются, выводится сообщение. При успехе запись сохраняется, модальное окно закрывается, реестр обновляется.

### Форма 3: Карточка приёма

**Статус**: НОВАЯ
**Связанная история**: Пользовательская история 4

**Элементы управления**:

| Элемент | Тип | Обязательный | Описание |
|---------|-----|:-----------:|----------|
| Блок «Общая информация» | Секция-контейнер | Да | Содержит: дата, ФИО врача, специальность, клиника, основные результаты, оценка врача, скан заключения |
| Блок «Препараты» | Секция-контейнер | Да | Содержит многострочное текстовое поле «Назначенные препараты» |
| Блок «Назначения» | Секция-контейнер | Да | Содержит многострочное текстовое поле «Другие процедуры и исследования» |
| Ссылка «Предыдущий приём» | Ссылка | Нет | Переход к предыдущему приёму того же врача (ФИО + специальность) |
| Ссылка «Следующий приём» | Ссылка | Нет | Переход к следующему приёму того же врача (ФИО + специальность) |
| Кнопка «Редактировать» | Кнопка | Да | Открывает модальное окно редактирования |
| Кнопка «Удалить» | Кнопка | Да | Удаляет запись с подтверждением |
| Кнопка «Закрыть» | Кнопка | Да | Возврат в реестр |

**Логика работы**:

При открытии информация отображается тремя визуальными блоками:
1. **«Общая информация»** — дата, ФИО врача, специальность, клиника, основные результаты приёма, оценка врача (звёзды), скан заключения (изображение).
2. **«Препараты»** — многострочное текстовое поле с назначенными препаратами.
3. **«Назначения»** — многострочное текстовое поле с другими процедурами и исследованиями.

Блоки «Препараты» и «Назначения» спроектированы с расчётом на будущее расширение (добавление дополнительных полей и элементов). Если поле блока пустое — блок отображается с пометкой «Не указано». Ссылки на предыдущий/следующий приём отображаются только если существуют другие приёмы того же врача (совпадение по ФИО и специальности).

## Требования *(обязательно)*

### Функциональные требования

- **FR-001**: Система ДОЛЖНА позволять пользователю создавать запись о приёме с полями: дата, ФИО врача, специальность, клиника, основные результаты, назначенные препараты, другие процедуры и исследования, скан заключения
- **FR-002**: Система ДОЛЖНА требовать заполнения полей «Дата», «ФИО врача» и «Специальность» при создании или редактировании записи
- **FR-003**: Система ДОЛЖНА позволять пользователю оценить врача и качество приёма по 5-балльной шкале (от 1 до 5)
- **FR-004**: Система ДОЛЖНА предоставлять отдельный раздел меню «Приёмы» с плиточным отображением записей
- **FR-005**: Система ДОЛЖНА отображать записи в реестре, отсортированные по дате приёма (новые сверху), по умолчанию — все записи
- **FR-006**: Система ДОЛЖНА позволять фильтровать записи по периоду дат и по специальности врача
- **FR-007**: Система ДОЛЖНА позволять пользователю загрузить изображение скана заключения и отправить его в LLM для автоматического распознавания и заполнения полей формы
- **FR-008**: Система ДОЛЖНА позволять пользователю проверить и отредактировать данные, распознанные LLM, перед сохранением записи
- **FR-009**: Система ДОЛЖНА позволять пользователю редактировать существующие записи о приёмах через модальное окно
- **FR-010**: Система ДОЛЖНА позволять пользователю удалять записи о приёмах с запросом подтверждения
- **FR-011**: Система ДОЛЖНА отображать карточку приёма с полной информацией по клику на плитку в реестре
- **FR-012**: Система ДОЛЖНА отображать в карточке приёма ссылки на предыдущий и следующий приём того же врача (определяется по совпадению ФИО и специальности), если они существуют
- **FR-013**: Система ДОЛЖНА отображать индикатор процесса во время LLM-распознавания скана
- **FR-014**: Система ДОЛЖНА отображать понятное сообщение об ошибке, если LLM-распознавание завершилось неудачей (сетевая ошибка, нераспознанный файл)
- **FR-015**: Система ДОЛЖНА сохранять скан заключения в локальном файловом хранилище приложения (путь к файлу указывается в записи о приёме)
- **FR-016**: Система ДОЛЖНА обеспечивать интеграцию с LLM API (OpenAI-совместимый REST API) для распознавания медицинских сканов в рамках данной задачи. Конфигурация провайдера (URL, ключ) должна выноситься через переменные окружения для возможности замены без изменения кода. LLM-провайдер может быть локальным (например, Ollama) или удалённым

### Нефункциональные требования

- **NFR-001**: Файлы сканов должны храниться вне публичного (доступного из браузера) каталога приложения
- **NFR-002**: Данные приёма должны быть изолированы — пользователь видит только собственные записи
- **NFR-003**: Система ДОЛЖНА поддерживать чтение существующих записей и ручной ввод новых без подключения к интернету (полностью оффлайн-режим). LLM-распознавание сканов доступно только при работающем LLM-провайдере (локальном или удалённом)

### Ключевые сущности

- **Приём**: Запись о приёме врача. Атрибуты: дата приёма, ФИО врача, специальность врача, клиника, основные результаты (текст), назначенные препараты (текст), другие процедуры и исследования (текст), скан заключения (изображение), оценка врача (число 1–5).
- **Специальность врача**: Текстовое значение, связанное с записью о приёме. Используется для фильтрации в реестре и для навигации между приёмами одного врача.

## Допущения

- Один пользователь системы — один пациент, ведущий собственную историю приёмов
- Данные хранятся локально в приложении (без облачной синхронизации в рамках данной функции)
- Основной формат скана — PDF, но система должна поддерживать и другие распространённые форматы изображений (JPEG, PNG, WebP)
- Специальности врачей не ограничены предопределённым списком — пользователь вводит текст свободно (фильтр по специальности строится по уникальным значениям из существующих записей)
- Оценка врача относится к конкретному приёму, а не к врачу в целом
- Максимальный размер загружаемого файла определяется стандартными ограничениями проекта

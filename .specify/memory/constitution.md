# Конституция my-health

## Основные принципы

### I. Конфиденциальность и безопасность данных

Все данные о здоровье пользователя являются конфиденциальной информацией. Система ДОЛЖНА:

- Хранить все данные исключительно локально — никакая телеметрия, аналитика или данные не отправляются на внешние серверы без явного действия пользователя
- Не требовать сетевой активности для основной функциональности (ввод, просмотр, редактирование данных)
- Предоставлять пользователю полный контроль над данными: экспорт, импорт, удаление
- Не кэшировать и не логировать чувствительные данные (результаты анализов, диагнозы, препараты) в системных логах или crash-отчётах

**Обоснование**: Медицинские данные подпадают под строгие нормы конфиденциальности. Однопользовательская модель упрощает задачу, но не отменяет ответственности.

### II. Локальное хранение, локальная обработка

Приложение работает полностью локально. Все данные хранятся в JSON-файлах файловой системы, сканы документов — в отдельных директориях.

- Структура JSON-файлов ДОЛЖНА быть стабильной и версионированной (поле `schemaVersion` в каждом файле)
- Пути к данным, конфигурации и сканам определяются через конфигурационный файл, а не захардкожены
- Чтение и запись файлов ДОЛЖНЫ быть атомарными (запись во временный файл → переименование) для предотвращения потери данных при сбоях
- Приложение ДОЛЖНО корректно обрабатывать отсутствие, повреждение или несовместимую версию файлов данных

**Обоснование**: Локальное хранение — ключевое архитектурное решение. Надёжность файловой работы критична для доверия пользователя.

### III. Типобезопасность и валидация

Весь код на TypeScript ДОЛЖЕН быть строго типизирован. Данные из файлов ДОЛЖНЫ валидироваться при загрузке.

- Запрещён `any` без явного обоснования в комментарии
- Все внешние данные (из JSON-файлов, от Tauri IPC) проходят валидацию через схемы (zod или аналог) до попадания в бизнес-логику
- Ошибки валидации ДОЛЖНЫ быть понятны пользователю (какой файл, какое поле, что ожидалось)
- Типы для UI-компонентов (props, state) ДОЛЖНЫ быть строго определены

**Обоснование**: Медицинские данные не терпят молчаливых ошибок типов. Валидация на границе ввода/вывода предотвращает каскадные сбои.

### IV. Модульность и переиспользование

Каждая значимая единица функциональности (сервис данных, компонент UI, утилита) оформляется как отдельный модуль.

- Один файл — одна основная ответственность (класс, сервис, компонент)
- Максимальный размер файла: 300 строк (без тестов). Превышение требует рефакторинга
- Общие утилиты и константы выносятся в отдельные модули, не дублируются
- Компоненты UI ДОЛЖНЫ быть максимально stateless; состояние поднимается к ближайшему общему предку

**Обоснование**: Предотвращает разрастание файлов («файлы-простыни»), упрощает тестирование и навигацию по коду.

### V. Простота и прагматизм

Начинать с минимального рабочего решения. Усложнение ДОЛЖНО быть обосновано реальной потребностью, а не гипотетическими сценариями.

- KISS: выбирать самое простое решение, которое работает правильно
- DRY: не дублировать код — выносить общее в функции/утилиты
- SOLID: соблюдать принципы проектирования при создании сервисов и компонентов
- Никаких «магических констант» — все значения, зависящие от контекста (лимиты, пороги, URL, пути), выносятся в конфигурацию
- Не раздувать файлы (см. принцип IV) и не раздувать зависимости — каждая новая зависимость требует обоснования

**Обоснование**: Однопользовательское десктопное приложение не нуждается в сложной архитектуре. Простота — залог поддерживаемости.

### VI. Наблюдаемость и отладочное логирование

Приложение ДОЛЖНО поддерживать два режима логирования, переключаемых через конфигурационный файл:

- **Обычный режим** — только ошибки и критичные предупреждения; логи минимальны
- **Отладочный режим** (`debug: true` в конфигурации) — детальное логирование в текстовые файлы:
  - Вход/выход каждой операции с данными (чтение, запись, обновление JSON-файлов)
  - IPC-вызовы между Tauri и frontend (метод, параметры, результат, длительность)
  - Ошибки валидации с контекстом (файл, поле, ожидаемый/фактический тип)
  - Инициализация и жизненный цикл сервисов
  - Действия пользователя (открытие форм, сохранение, удаление)
- Лог-файлы хранятся локально, имеют ограничение по размеру (ротация по достижении лимита)
- Отладочный режим НЕ ДОЛЖЕН логировать чувствительные медицинские данные (значения анализов, диагнозы, названия препаратов) — вместо этого логируются только факт операции и метаданные
- Формат логов: структурированный текст с меткой времени, уровнем, модулем и сообщением

**Обоснование**: Без детальных логов отладка проблем с данными затруднительна. При этом логи не должны превращаться в утечку конфиденциальной информации.

## Технологический стек

| Слой | Технология |
|------|-----------|
| Runtime | Tauri (desktop) |
| Frontend | React + TypeScript |
| Стили | Tailwind CSS + shadcn/ui |
| Хранение данных | JSON-файлы в файловой системе |
| Сканы документов | Файловая система (изображения, PDF) |
| Платформы | Windows, Linux, macOS |

- Все константы и конфигурационные параметры выносятся в отдельные конфигурационные файлы
- Структура JSON-файлов документируется в формате JSON Schema

## Требования к пользовательскому интерфейсу

Приложение работает только на десктопе (Windows, Linux, macOS). Макетов и дизайн-системы отдельно не существует — компоненты проектируются агентом в соответствии с референсным стилем.

- **Референсный стиль**: Vercel Dashboard — современный, светлый, минималистичный; карточная компоновка, мягкие тени, серо-белая палитра с акцентами на ключевых действиях
- **Цветовая схема**: только светлая тема; нейтральный фон (белый/светло-серый), текстовые акценты для интерактивных элементов, цветовые индикаторы для статусов (норма/отклонение/тревога)
- **Типографика**: системные шрифты платформы (Segoe UI / SF Pro / Cantarell), без кастомных шрифтов; числовые данные — `tabular-nums` для выравнивания по столбцам
- **Компоновка**: карточки для группировки данных, таблицы для списков записей, формы с вертикальной раскладкой полей; минимум визуального шума — данные важнее декораций
- **Компоненты**: использовать shadcn/ui как базовую библиотеку; кастомизация через CSS-переменные Tailwind, а не через переопределение стилей компонентов
- **Адаптивность**: поддержка стандартных десктопных разрешений (от 1280×720); окно может ресайзиться, контент не должен ломаться

## Стандарты качества кода

- **Линтинг и форматирование**: ESLint + Prettier (или эквиваленты для выбранного стека), настроенные на строгий режим
- **TypeScript**: `strict: true` в `tsconfig.json`, без исключений
- **Тесты**: Критическая бизнес-логика (парсинг данных, расчёты, валидация) покрывается unit-тестами
- **Именование**: Осмысленные имена переменных, функций, компонентов. Без аббревиатур кроме общепринятых

## Управление

- Настоящая конституция имеет приоритет над всеми остальными практиками и рекомендациями в проекте
- Любое изменение в принципах требует:
  1. Описания причины изменения
  2. Обновления версии (семантическое версионирование)
  3. Обновления даты последнего изменения
  4. Проверки согласованности с шаблонами (`plan-template.md`, `spec-template.md`, `tasks-template.md`)
- Версионирование конституции: `MAJOR.MINOR.PATCH`
  - MAJOR — удаление или обратная несовместимость принципов
  - MINOR — добавление нового принципа или раздела
  - PATCH — уточнения, формулировки, исправления
- Все PR и ревью кода ДОЛЖНЫ проверять соответствие принципам конституции
- Если возникает необходимость нарушить принцип — это документируется в разделе «Отслеживание сложности» плана реализации с обоснованием

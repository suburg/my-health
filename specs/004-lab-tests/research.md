# Research: Модуль «Анализы»

**Дата**: 2026-04-10
**Ветка**: `004-lab-tests`

## Обзор

Все пункты NEEDS CLARIFICATION были разрешены в процессе `/speckit.clarify` до начала планирования. Данный документ фиксирует принятые решения и результаты исследования.

---

## Decision 1: Механизм ручной привязки показателей

**Решение**: Диалог выбора из справочника (поиск + выбор эталонного названия). Полная форма управления справочником — отдельная задача в будущем.

**Обоснование**:
- Согласуется с принципом KISS (конституция, принцип V)
- Минимальный рабочий механизм: пользователь кликает на «не распознан» → открывается диалог с поиском по справочнику → выбирает эталонное название → привязка сохраняется
- Полная CRUD-форма для справочника (создание показателей, редактирование синонимов) выходит за рамки MVP и будет вынесена в отдельную задачу

**Рассмотренные альтернативы**:
- Отдельная форма управления справочником — отвергнута как избыточная для текущей версии
- Только LLM без ручной привязки — отвергнута, так как LLM может ошибаться и пользователю нужен контроль

## Decision 2: Размер справочника и передача в LLM-контекст

**Решение**: Справочник до ~200 записей, передаётся целиком в системном промпте при каждом распознавании.

**Обоснование**:
- ~200 записей покрывают основные показатели общего анализа крови, мочи, биохимии, гормонов
- Размер контекста для 200 записей (~10-15KB JSON) помещается в системный промпт большинства LLM
- Простая архитектура — не требуется предфильтрация или кеширование контекста
- При каждом распознавании справочник актуален (включая пользовательские привязки)

**Рассмотренные альтернативы**:
- Фильтрация релевантной части по ключевым словам — отвергнута как избыточная для ≤200 записей
- Кеширование контекста между сессиями — отвергнуто, так как добавляет сложность без значимой выгоды

## Decision 3: Поиск предыдущего значения показателя

**Решение**: По эталонному названию, только в рамках того же типа анализа.

**Обоснование**:
- Этaлонное название обеспечивает однозначное сопоставление между анализами
- Ограничение типом анализа исключает бессмысленные сравнения (напр., «Гемоглобин» из крови ≠ «Гемоглобин» из мочи)
- Простая логика: `SELECT * FROM lab_tests WHERE type = current.type AND date < current.date ORDER BY date DESC LIMIT 1`

**Рассмотренные альтернативы**:
- Поиск среди всех анализов независимо от типа — отвергнут, даёт ложные соответствия
- Строгое соответствие по типу + лаборатории — отвергнуто как излишне ограничительное (пользователь может сдавать один тип анализа в разных лабораториях)

## Decision 4: Архитектура таблицы показателей

**Решение**: Динамическая таблица с контекстными колонками — числовые и текстовые показатели обрабатываются единообразно, но с разными типами ввода и валидации.

**Обоснование**:
- Большинство показателей — числовые, справочник по умолчанию содержит их
- Нечисловые показатели (цвет, прозрачность, реакция) — поддерживаются через текстовый тип с допустимыми значениями
- Контекстная колонка «Референс» меняет содержимое в зависимости от типа: интервал для числовых, список допустимых для текстовых

**Рассмотренные альтернативы**:
- Раздельные таблицы для числовых и текстовых — отвергнута как усложнение UX
- Только числовые показатели — отвергнута, так как общий анализ мочи содержит текстовые показатели

---

## Decision 5: Переиспользование компонентов doctor-visits

**Решение**: Переиспользовать `ScanUploader` из модуля doctor-visits. Остальные компоненты — новые, специфичные для lab-tests.

**Обоснование**:
- `ScanUploader` универсален — загрузка изображений/PDF + LLM-распознавание
- Остальные компоненты doctor-visits слишком специфичны для приёмов (VisitCard, VisitTile, VisitModal имеют другую структуру данных)
- Следует принцип DRY — не дублировать код ScanUploader

---

## Decision 6: Формат справочника показателей

**Решение**: JSON-файл `src/config/indicator-reference.json` со структурой, аналогичной LabTestIndicatorReference.

**Обоснование**:
- JSON — нативный формат для проекта, легко загружается на frontend
- Файл в `src/config/` — стандартное место для встроенных данных
- Структура совпадает с сущностью LabTestIndicatorReference из спецификации
- Может быть расширен пользователем в будущем (запись в пользовательскую директорию)

**Структура записи справочника**:
```json
{
  "canonicalName": "Гемоглобин",
  "synonyms": ["HGB", "Hemoglobin", "Hb", "Гемоглобин (Hb)"],
  "valueType": "numeric",
  "testTypes": ["blood", "biochemistry"],
  "unit": "г/л",
  "referenceType": "interval",
  "typicalReference": { "min": 120, "max": 160 }
}
```
Для текстовых:
```json
{
  "canonicalName": "Цвет мочи",
  "synonyms": ["Color", "Цвет"],
  "valueType": "textual",
  "testTypes": ["urine"],
  "unit": null,
  "referenceType": "list",
  "allowedValues": ["соломенно-жёлтый", "жёлтый", "янтарный", "бесцветный"]
}
```

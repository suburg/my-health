# Исследование: Ведение информации о приёмах врача

## RQ-001: Стратегия хранения сканов (NC-001)

**Вопрос**: Копировать загруженный файл в AppData или хранить путь к оригиналу?

### Решение
**Копировать файл в директорию `scans/` внутри AppData.**

### Обоснование
- Соответствует принципу «локальное хранение» — файл становится частью данных приложения
- Пользователь может удалить оригинал — данные не потеряются
- Упрощает экспорт/резервное копирование — всё в одном месте
- Путь к файлу сохраняется в записи JSON как относительный путь (`scans/<uuid>.jpg`)
- UUID предотвращает коллизии имён

### Рассмотренные альтернативы
1. **Хранение пути к оригиналу** — ненадёжно, пользователь может переместить/удалить файл
2. **Base64 в JSON** — раздует JSON-файл, усложнит работу с большими файлами

## RQ-002: Паттерн IPC-вызова для LLM-распознавания (NC-002)

**Вопрос**: Синхронный или асинхронный IPC-вызов для LLM?

### Решение
**Синхронный IPC-вызов (Tauri `invoke`), но UI обрабатывает его асинхронно.**

### Обоснование
- Tauri `invoke` возвращает Promise — на стороне UI это асинхронная операция
- На стороне Rust команда блокирующая, но выполняется в отдельном потоке (Tauri автоматически раннинг задачи в tokio/spawn)
- Для пользовательского сценария: пользователь нажал кнопку → спиннер → результат. Это естественно для десктопного приложения
- Таймаут на стороне Rust (например, 60 секунд) предотвращает бесконечное ожидание
- Альтернатива с event-driven подходом (Tauri events) избыточна для одного пользователя

### Рассмотренные альтернативы
1. **Tauri Events (emit/listen)** — сложнее, нужен для long-running фоновых задач; наш сценарий — foreground-операция
2. **Web Worker + fetch** — невозможен в Tauri напрямую, всё через IPC

## RQ-003: Поддержка PDF (NC-003)

**Вопрос**: Как обрабатывать PDF-сканы?

### Решение
**Полная поддержка PDF: хранение, отображение превью, конвертация в изображение для LLM.**

### Обоснование
- PDF — основной формат сканов медицинских заключений (экспорт из медицинских информационных систем)
- Конвертация всех страниц PDF → PNG выполняется на стороне Rust с помощью `pdfium-render` (биндинг к PDFium — движку Chrome)
- Для LLM отправляются конвертированные изображения всех страниц PDF одним запросом
- Для отображения превью в UI используется PNG-превью первой страницы
- PDF-файл сохраняется как есть в директорию `scans/`

### Архитектура обработки PDF

```
PDF загружен → имя файла: ГГГГ_ММ_ДД.Специальность.uuid.pdf → scans/
            → pdfium-render рендерит ВСЕ страницы → scans/<business-name>_p1.png, _p2.png, ...
            → превью первой страницы показывается в UI
            → ВСЕ страницы (PNG) отправляются в LLM в одном запросе (массив image_url)
```

Формат имени файла: `<date>_<specialty-sanitized>_<uuid>.pdf`
- `date` — дата приёма в формате YYYY_MM_DD (если дата не распознана — `0000_00_00`)
- `specialty-sanitized` — специальность: кириллица и латиница сохраняются, пробелы → `_`, спецсимволы удаляются
- `uuid` — короткие 8 символов UUID для уникальности

Пример: `2026_04_09_Кардиолог_a1b2c3d4.pdf`

Для файлов изображений: аналогичный формат — `2026_04_09_Кардиолог_a1b2c3d4.png`

Для страниц-превью PDF: `2026_04_09_Кардиолог_a1b2c3d4_p1.png`, `_p2.png`, ...

Для LLM все страницы объединяются в один запрос — OpenAI Vision API поддерживает массив изображений в одном сообщении. Это сохраняет контекст документа целиком.

### Рассмотренные альтернативы
1. **Все страницы в одном запросе** (выбрано) — OpenAI Vision API поддерживает массив `image_url`. Сохраняет полный контекст документа.
2. **Только первая страница** — отклонено: критичные данные (заключения, назначения) часто на последних страницах.
3. **По одной странице на запрос** — отклонено: потеря контекста, множественные API-вызовы, сложно агрегировать результат.
4. **`pdfium-render`** (рендеринг) — биндинг к PDFium, один файл библиотеки, хорошая производительность
5. **`poppler`** — тяжелее, сложнее деплой на Windows
6. **`mupdf`** — лицензионные ограничения (AGPL)
7. **`pdf-extract`** (чистый Rust) — пока не поддерживает рендеринг в изображения

### Зависимости

**Rust**: `pdfium-render` (крейт). Требует PDFium binary — поставляется с приложением.
**Frontend**: новых зависимостей нет — превью отображается как обычное `<img>` с PNG-файлом.

### Итог
Поддерживаемые форматы: `image/jpeg`, `image/png`, `image/webp`, `image/gif`, `image/bmp`, `application/pdf`.
При загрузке PDF: все страницы рендерятся в PNG, первая страница — превью в UI, все страницы отправляются в LLM.

## RQ-004: Структура данных — отдельный JSON-файл (NC-004)

**Вопрос**: Отдельный файл или расширение существующего?

### Решение
**Отдельный файл `doctor-visits.json`.**

### Обоснование
- Следует существующему паттерну: `profile.json`, `health.json`, `metric-config.json` — каждый домен в своём файле
- Изоляция данных — проще бэкап/экспорт/отладка
- Независимая эволюция схемы — изменения в doctor-visits не затрагивают health

### Структура файла
```json
{
  "schemaVersion": 1,
  "visits": [
    {
      "id": "uuid-v4",
      "date": "2026-04-09",
      "doctorName": "Иванов Иван Иванович",
      "specialty": "Кардиолог",
      "clinic": "Поликлиника №1",
      "results": "...",
      "medications": "...",
      "procedures": "...",
      "scanPath": "scans/<uuid>.jpg",
      "rating": 4,
      "createdAt": "2026-04-09T10:00:00Z",
      "updatedAt": "2026-04-09T10:00:00Z"
    }
  ]
}
```

## RQ-005: Промпт для LLM-распознавания медицинских документов (NC-005)

**Вопрос**: Какая системная подсказка даёт наилучшие результаты?

### Решение
**Структурированный JSON-промпт с примером ответа. Хранится в конфигурационном файле `llm-prompt.json` в директории данных приложения — меняется без пересборки.**

### Расположение промпта

Файл: `%APPDATA%\com.myhealth.app\llm-prompt.json` (рядом с остальными данными)

```json
{
  "schemaVersion": 1,
  "systemPrompt": "Ты — ассистент для распознавания медицинских документов...",
  "exampleResponse": {
    "doctorName": "Иванов Иван Иванович",
    "specialty": "Кардиолог",
    "clinic": "Городская поликлиника №1",
    "date": "2026-04-01",
    "results": "Жалобы на повышение АД до 150/95. Объективно: ЧСС 78 уд/мин. Рекомендовано: ЭКГ, общий анализ крови, консультация нефролога.",
    "medications": "Эналаприл 5мг — 1 раз в день утром\nАмлодипин 5мг — 1 раз в день вечером",
    "procedures": "ЭКГ — через 2 недели\nОбщий анализ крови — натощак\nХолтеровское мониторирование — запись на 24ч"
  }
}
```

Промпт загружается при старте приложения, кэшируется в памяти. Перезачитывается при каждом вызове распознавания (файл маленький — 0.5мс overhead).

### Промпт (системное сообщение)
```
Ты — ассистент для распознавания медицинских документов. 
Проанализируй изображение(я) медицинского документа (заключение врача, рецепт, направление, выписка и т.п.) 
и извлеки информацию в формате JSON.

ВНИМАНИЕ: Тебе может быть передано несколько изображений — это страницы одного документа.
Анализируй все страницы совместно, извлекая полную информацию из всего документа.

Ожидаемые поля:
- doctorName: ФИО врача (строка, если не распознано — null)
- specialty: Специальность врача (строка, если не распознано — null)
- clinic: Название клиники/учреждения (строка, если не распознано — null)
- date: Дата приёма в формате YYYY-MM-DD (строка, если не распознано — null)
- results: Основные результаты/заключение врача (строка, если не распознано — null)
- medications: Назначенные препараты/лекарства (строка, если не распознано — null)
- procedures: Назначенные процедуры и исследования (строка, если не распознано — null)

Вот пример корректного ответа для ориентира:
{exampleResponse}

Ответь ТОЛЬКО валидным JSON объектом без дополнительного текста, markdown-обрамления или комментариев.
Если поле не удалось распознать — укажи null.
```

При загрузке `{exampleResponse}` подставляется из `llm-prompt.json`.

### Формат запроса к OpenAI-совместимому API
```json
{
  "model": "gpt-4o",
  "messages": [
    {
      "role": "system",
      "content": "..."
    },
    {
      "role": "user",
      "content": [
        {
          "type": "image_url",
          "image_url": {
            "url": "data:image/jpeg;base64,<page1_base64>",
            "detail": "high"
          }
        },
        {
          "type": "image_url",
          "image_url": {
            "url": "data:image/png;base64,<page2_base64>",
            "detail": "high"
          }
        }
      ]
    }
  ],
  "max_tokens": 1000,
  "response_format": { "type": "json_object" }
}
```

Для изображений — один элемент массива. Для PDF — все страницы как отдельные `image_url` элементы в одном сообщении. OpenAI Vision API обрабатывает все изображения совместно, сохраняя контекст документа.

### Обоснование
- `response_format: json_object` гарантирует валидный JSON ответ
- Base64-кодирование изображения — стандартный подход для OpenAI Vision API
- `detail: high` — для медицинских документов важно высокое качество распознавания
- Системный промпт явно описывает все ожидаемые поля

### Рассмотренные альтернативы
1. **Function Calling** — требует поддержки у провайдера, менее универсально
2. **Своя модель OCR + парсинг** — слишком сложно для MVP
3. **Простой текстовый промпт без JSON** — сложнее парсить ответ

### Переменные окружения для конфигурации
```
LLM_API_URL=https://api.openai.com/v1
LLM_API_KEY=sk-...
LLM_MODEL=gpt-4o
```
Для Ollama (локальный):
```
LLM_API_URL=http://localhost:11434/v1
LLM_API_KEY=unused
LLM_MODEL=llava
```

## RQ-006: Паттерн маршрутизации для нового экрана

**Вопрос**: Как интегрировать экран «Приёмы» в существующую навигацию?

### Решение
**Добавить новый screen type в существующий state-based router в `App.tsx`.**

### Обоснование
- Следует существующему паттерну: `useState<Screen>` с экранными компонентами
- Добавляется `currentScreen === "doctorVisits"` 
- В `NavMenu` добавляется пункт «Приёмы» с иконкой (lucide-react: `Stethoscope` или `ClipboardList`)
- Не требует внешних роутеров — сохраняет простоту

## RQ-007: Лучшие практики для плиточного интерфейса

**Вопрос**: Как реализовать плиточный реестр с фильтрацией?

### Решение
**CSS Grid с карточками shadcn/ui, фильтрация на стороне frontend.**

### Обоснование
- Данные загружаются полностью (небольшой объём), фильтрация — на клиенте
- CSS Grid обеспечивает адаптивную плиточную раскладку
- shadcn/ui `Card` компонент — базовый строительный блок
- lucide-react иконки для визуальных акцентов на плитках

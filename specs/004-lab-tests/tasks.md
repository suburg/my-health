# Задачи: Модуль «Анализы»

**Входные данные**: Проектная документация из `/specs/004-lab-tests/`
**Предварительные требования**: plan.md, spec.md, data-model.md, contracts/, research.md

**Тесты**: Включены unit-тесты для критической бизнес-логики (парсинг данных, валидация, сравнение показателей, поиск предыдущего значения) согласно конституции проекта.

**Организация**: Задачи сгруппированы по пользовательским историям, чтобы обеспечить независимую реализацию и тестирование каждой истории.

## Формат: `[ID] [P?] [Story] Описание`

- **[P]**: Можно запускать параллельно (разные файлы, нет зависимостей)
- **[Story]**: Какой пользовательской истории принадлежит задача (например, US1, US2, US3)
- Включайте точные пути к файлам в описаниях

## Соглашения о путях

Один проект (Tauri + React):
- Frontend: `src/`
- Backend (Rust/Tauri): `src-tauri/src/`
- Типы: `src/types/`
- Тесты: unit-тесты рядом с исходными файлами (`__tests__/`)

## Зависимости и порядок выполнения

### Зависимости фаз

- **Настройка (Фаза 1)**: Нет зависимостей — можно начинать немедленно
- **Фундаментальные (Фаза 2)**: Зависит от завершения Настройки — **БЛОКИРУЕТ** все пользовательские истории
- **Пользовательские истории (Фаза 3+)**: Все зависят от завершения Фундаментальной фазы
  - US1 (P1) → US2 (P2) → US3 (P3) → US4 (P4) → US5 (P5) — последовательная рекомендуемая выдача
  - После Фазы 2 все US могут разрабатываться параллельно
- **Полировка (Фаза 8)**: Зависит от завершения всех пользовательских историй

| История | Зависит от | Может начаться после |
|---------|-----------|---------------------|

## Стратегия реализации

### Сначала MVP (US1 — Реестр)

1. Завершить Фазу 1: Настройка
2. Завершить Фазу 2: Фундаментальные (КРИТИЧНО — блокирует все истории)
3. Завершить Фазу 3: Пользовательская история 1
4. **ОСТАНОВИТЬСЯ и ПРОВЕРИТЬ**: Реестр загружается, сортирует, фильтрует
5. Развернуть/продемонстрировать

### Инкрементальная выдача

1. Фаза 1 + Фаза 2 → CRUD и хранилище готовы
2. + US1 → Реестр работает → MVP!
3. + US2 → Ручной ввод с показателями → Демо
4. + US3 → LLM-распознавание → Демо
5. + US4 → Эталонные названия и привязка → Демо
6. + US5 → Детальная карточка с историей → Демо
7. Фаза 8 → Полировка, тесты, проверка quickstart

### Стратегия параллельной команды

С несколькими разработчиками:

1. Команда завершает Настройку + Фундаментальные вместе
2. После завершения Фундаментальных:
   - Разработчик A: US1 (Реестр) → US5 (Карточка)
   - Разработчик B: US2 (Ввод) → US3 (LLM)
   - Разработчик C: US4 (Эталонные названия)
3. Истории завершаются и интегрируются независимо

**Дата**: 2026-04-07
**Ветка**: `001-app-scaffold-auth`

## R-001: Хэширование пин-кода в Tauri-приложении

**Decision**: Использовать argon2id через Rust-крейт `argon2` в Tauri backend. Пин-код (строка цифр) хэшируется на стороне Rust, хэш хранится в JSON-файле профиля.

**Rationale**: 
- Пин-код обрабатывается в Tauri backend (Rust), не в JavaScript — исключена утечка в DevTools
- argon2id — рекомендованный алгоритм для хэширования паролей (OWASP 2024)
- Для 4–6-значного пин-кода важно использовать параметры с высокой стоимостью памяти и вычислений, чтобы затруднить перебор
- Rust-крейт `argon2` стабилен, хорошо поддерживается

**Alternatives considered**:
- bcrypt — устаревший для новых проектов, argon2 предпочтительнее
- SHA-256 + salt — недостаточно для коротких пин-кодов (малое пространство ключей)
- Хэширование на JS (frontend) — небезопасно, хэш можно перехватить в DevTools

## R-002: Атомарная запись JSON-файлов

**Decision**: Запись через временный файл + `std::fs::rename` (атомарная замена на POSIX и Windows). Библиотека `atomic-writefile` или ручная реализация.

**Rationale**:
- Предотвращает потерю данных при сбое питания или крахе во время записи
- `rename` атомарен на всех целевых платформах (Windows, Linux, macOS)
- Требует минимального кода — KISS

**Alternatives considered**:
- SQLite — переусложнение для одного профиля
- Прямая запись без атомарности — риск повреждения данных при сбое

## R-003: Валидация данных на frontend

**Decision**: Использовать zod для схем валидации на React-frontend. Схемы объявляются в `src/lib/validations.ts` и переиспользуются между формами.

**Rationale**:
- Zod — стандарт де-факто для TypeScript-валидации в React-экосистеме
- Автоматический вывод типов из схем (z.infer) — DRY
- Интеграция с React Hook Form через `@hookform/resolvers` — стандартный паттерн для shadcn/ui
- Схемы можно дублировать на Rust-стороне для двойной валидации (принцип III конституции)

**Alternatives considered**:
- yup — менее популярен в TS-проектах
- Ручная валидация — больше кода, выше риск ошибок

## R-004: Состояние аутентификации в React

**Decision**: Простой React Context + useReducer для хранения состояния isAuthenticated. Без внешних библиотек state management.

**Rationale**:
- Однопользовательское приложение с бинарным состоянием (вошёл/не вошёл) — KISS
- React Context достаточно для одной сущности
- Избегает зависимостей (Redux, Zustand) — принцип V конституции

**Alternatives considered**:
- Zustand — хорошая библиотека, но избыточна для одного флага
- Redux — явное переусложнение

## R-005: Makefile для кроссплатформенной сборки Tauri

**Decision**: Makefile с целями `install`, `dev`, `build`, `clean`. На Windows пользователи используют `nmake` или `mingw32-make`. Альтернативно — npm-скрипты в `package.json`.

**Rationale**:
- Makefile — стандартный инструмент сборки для Unix-систем
- Tauri CLI предоставляет `tauri dev` и `tauri build` — Makefile обёртывает их
- Дублирование в package.json для Windows-пользователей без make

**Alternatives considered**:
- Только package.json — не соответствует требованию FR-011 (Makefile)
- Justfile — менее распространён, чем make

## R-006: Логирование в отладочном режиме

**Decision**: Простой логгер на TypeScript, записывающий в файл через Tauri `fs` API. Формат: `[TIMESTAMP] [LEVEL] [MODULE] message`. Конфигурация `debug: true` в JSON-конфиге.

**Rationale**:
- Минимальная реализация — KISS
- Текстовый файл — легко читать и анализировать
- Tauri `fs` API обеспечивает доступ к файловой системе
- Логи не содержат чувствительных данных (принцип I и VI конституции)

**Alternatives considered**:
- winston/pino — избыточны, требуют npm-зависимостей
- Консольный лог только — не сохраняется между сеансами

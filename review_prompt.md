# Промпт для ИИ-ревью кода проекта my-health

## Контекст проекта

**my-health** — десктопное приложение для ведения медицинских данных на **Tauri v2** (Rust backend + React 19 frontend на TypeScript). Код пишется ИИ-агентами и практически не проверяется человеком.

**Архитектура**: Двухуровневая — Frontend (`src/`) вызывает Rust-команды (`src-tauri/src/commands/`) через IPC (`invoke`). Данные хранятся в JSON-файлах в `appDataDir` с атомарной записью.

**Технологический стек**: Tauri v2, React 19, TypeScript ~5.8, Vite v7, Tailwind CSS v4, Zod v4, Recharts, Lucide React, Vitest v4, ESLint v9.

## Задача

Проведи полное ревью кода проекта. Найди баги, уязвимости, нарушения соглашений, «мусор» и потенциальные проблемы. Проект написан ИИ — обращай особое внимание на типичные ошибки генерации.

## Что проверять

### 1. Rust backend (`src-tauri/`)

- **Безопасность обработки данных**: валидация всех входных параметров команд, отсутствие паники (`unwrap()`, `expect()`) в продакшн-коде — использовать `?` и типизированные ошибки
- **Атомарность записи**: запись JSON через временный `.tmp` файл + `fs::rename`, а не напрямую
- **`#[serde(default)]`** на новых полях структур — обратная совместимость при чтении старых JSON-файлов
- **Управление файлами**: проверка существования, прав, корректная обработка ошибок при чтении/записи/удалении файлов сканов и аттачментов
- **Отсутствие утечек**: закрытие файлов, корректная обработка `Result`, нет `unwrap()` на пользовательских данных
- **thiserror**: все ошибки типизированы, сообщения на русском, нет «сырых» `String` ошибок в командах
- **Конкурентность**: `static Mutex` кэши — нет дедлоков, корректная обработка `PoisonError`
- **cargo check** проходит без ошибок и варнингов

### 2. TypeScript frontend (`src/`)

- **Нет `any`**: вся типизация строгая, `strict: true`
- **Zod v4 API**: корректное использование (`z.string().regex()`, `z.union()`, а не `.or()`)
- **Валидация на границе**: все данные от IPC проходят через Zod-схемы перед использованием
- **Обработка IPC-ошибок**: проверка `"error" in result` во всех вызовах `invoke()`
- **Отсутствие утечек памяти**: `useEffect` cleanup для подписок, интервалов, слушателей событий
- **React 19**: корректное использование хуков, нет нарушений правил (хуки только на верхнем уровне)
- **Ключи в списках**: `key` на уникальных ID, не на индексе массива
- **`noUnusedLocals` / `noUnusedParameters`**: нет неиспользуемых импортов, переменных, параметров
- **Файлы ≤ 300 строк** (без тестов) — нарушение требует рефакторинга

### 3. Типы и контракты

- **Согласованность типов**: TypeScript-типы (`src/types/index.ts`) соответствуют Rust-структурам (`serde(rename_all = "camelCase")`)
- **IPC-контракты**: поля в `invoke()` совпадают с `#[derive(Deserialize)]` в Rust (camelCase)
- **`schemaVersion`**: все JSON-файлы данных имеют поле `schemaVersion`, код корректно обрабатывает разные версии

### 4. UI и стили

- **Tailwind CSS v4**: корректный синтаксис (без устаревших `@apply` паттернов v3)
- **Доступность**: `aria-label` на кнопках без текста, `htmlFor` на label, клавиатурная навигация
- **Состояния**: загрузка, ошибка, пустое состояние — отображаются корректно
- **Нет дублирования**: одинаковые данные не рендерятся дважды (например, шапка карточки и обёртка страницы)

### 5. Логирование и конфиденциальность

- **Логи НЕ содержат медицинских данных**: в `logger` не попадают диагнозы, препараты, результаты приёмов
- **Два режима**: обычный (только ошибки) и отладочный (`debug: true`) — детальное логирование
- **MODULE_NAME**: каждый сервис/хук имеет строковую константу для логгера

### 6. Тесты

- **Только unit-тесты**: нет интеграционных и e2e тестов
- **Покрытие**: ключевые утилиты покрыты тестами (валидация, форматирование, фильтрация)
- **Моки**: IPC-вызовы замокированы через `vi.mock("@tauri-apps/api/core")`
- **`npm test`** проходит без ошибок

### 7. Сборка и линтинг

- **`npm run lint`** — 0 ошибок ESLint
- **`npm run build`** — `tsc` и `vite build` без ошибок
- **`cargo check`** — без ошибок и варнингов
- **`npm test`** — все тесты проходят

## Формат ответа

Для каждой найденной проблемы укажи:

1. **Файл и строка** (примерно)
2. **Тип**: `bug` / `security` / `convention` / `quality` / `performance`
3. **Серьёзность**: `critical` / `high` / `medium` / `low`
4. **Описание**: что не так и почему это проблема
5. **Рекомендация**: как исправить (с примером кода, если нужно)

В конце — сводку: количество проблем по типам и серьёзности, общий вердикт.

## Приоритеты

1. **Баги и уязвимости** — то, что сломает приложение или создаст угрозу данным
2. **Потеря данных** — неатомарная запись, отсутствие обработки ошибок при записи JSON
3. **Нарушения конституции** — `any`, файлы > 300 строк, отсутствие валидации
4. **Качество** — неиспользуемый код, дублирование, плохие имена
5. **Стиль** — форматирование, порядок импортов (менее критично, проверяется линтером)

## Не отмечать

- Проблемы форматирования, которые исправит Prettier
- Предложения рефакторинга, если код работает и соответствует соглашениям
- Вопросы архитектуры, если текущее решение работает и не нарушает KISS
- Предложения добавить зависимости без явной необходимости

**Входные данные**: Проектная документация из `/specs/001-app-scaffold-auth/`
**Предварительные требования**: plan.md (обязательно), spec.md (обязательно), research.md, data-model.md, contracts/

**Тесты**: Да — спецификация включает тестирование пользовательских сценариев (unit + e2e).

**Организация**: Задачи сгруппированы по пользовательским историям, чтобы обеспечить независимую реализацию и тестирование каждой истории.

## Формат: `[ID] [P?] [Story] Описание`

- **[P]**: Можно запускать параллельно (разные файлы, нет зависимостей)
- **[Story]**: Какой пользовательской истории принадлежит задача (например, US1, US2, US3)
- Включайте точные пути к файлам в описаниях

## Зависимости и порядок выполнения

### Зависимости фаз

- **Настройка (Фаза 1)**: Нет зависимостей — можно начинать немедленно
- **Фундаментальные (Фаза 2)**: Зависит от завершения Настройки — БЛОКИРУЕТ все пользовательские истории
- **Пользовательские истории (Фаза 3+)**: Все зависят от завершения Фундаментальной фазы
  - US1 → US2 → US3 → US4 (последовательно, каждая добавляет функциональность)
- **Полировка (Фаза 7)**: Зависит от завершения всех пользовательских историй

### Зависимости пользовательских историй

- **US1 (P1)**: Может начаться после Фундаментальной (Фаза 2) — Нет зависимостей от других историй
- **US2 (P2)**: Зависит от US1 (нужен успешный вход для отображения основного экрана)
- **US3 (P3)**: Зависит от US2 (навигация из MainScreen к профилю)
- **US4 (P4)**: Зависит от US2 (кнопка «Выход» в меню MainScreen)

### Внутри каждой пользовательской истории

- Backend (IPC-команды) перед Frontend (сервисы и компоненты)
- Сервисы перед компонентами
- Компоненты перед интеграцией
- История завершена до перехода к следующему приоритету

### Возможности для параллелизма

- T002, T003, T004 (Настройка) — параллельно
- T006, T007, T008, T009, T010 (Фундаментальные) — параллельно (разные файлы)
- T013, T014, T015 (US1 backend) — параллельно
- T019, T020 (US1 frontend компоненты) — параллельно
- T024, T025, T027 (US2) — параллельно (разные файлы)
- T028, T029 (US3 backend) — параллельно

## Стратегия реализации

### Сначала MVP (только Пользовательская история 1)

1. Завершить Фазу 1: Настройка
2. Завершить Фазу 2: Фундаментальные (КРИТИЧНО — блокирует все истории)
3. Завершить Фазу 3: Пользовательская история 1
4. **ОСТАНОВИТЬСЯ и ПРОВЕРИТЬ**: Регистрация → вход → основной экран
5. Развернуть/продемонстрировать если готово

### Инкрементальная выдача

1. Настройка + Фундаментальные → Фундамент готов
2. US1 (регистрация + вход) → Проверить независимо → Демо (MVP!)
3. US2 (основной экран + меню) → Проверить независимо → Демо
4. US3 (профиль + смена пин-кода) → Проверить независимо → Демо
5. US4 (выход) → Проверить → Демо
6. Полировка → Финальное тестирование

## Примечания

- Задачи [P] = разные файлы, нет зависимостей
- Метка [Story] связывает задачу с конкретной пользовательской историей для отслеживания
- Каждая пользовательская история должна быть независимо завершаема и тестируема
- Фиксировать после каждой задачи или логической группы
- Останавливаться на любой контрольной точке для независимой проверки истории
- Избегать: неопределенных задач, конфликтов одних и тех же файлов, кросс-исторических зависимостей, нарушающих независимость

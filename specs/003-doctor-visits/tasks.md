## Зависимости и порядок выполнения

### Зависимости фаз

```
Фаза 1: Настройка (T001-T003)
    ↓
Фаза 2: Фундаментальные (T004-T013) — БЛОКИРУЕТ ВСЕ US
    ↓
Фаза 3: US1 — Ручное добавление (T014-T023) → MVP
    ↓
Фаза 4: US2 — Реестр с фильтрацией (T024-T025)
    ↓ (параллельно с US3)
Фаза 5: US3 — LLM распознавание (T026-T033)
    ↓ (параллельно с US4)
Фаза 6: US4 — Карточка и навигация (T034-T037)
    ↓
Фаза 7: US5 — Редактирование/удаление (T038-T042) — зависит от US4
    ↓
Фаза 8: Полировка (T043-T048)
```

### Зависимости пользовательских историй

- **US1 (P1)**: Может начаться после Фазы 2. Нет зависимостей от других историй.
- **US2 (P2)**: Может начаться после Фазы 2. Использует компоненты US1 (VisitRegistry, VisitTile).
- **US3 (P3)**: Может начаться после Фазы 2. Интегрируется с VisitModal из US1.
- **US4 (P4)**: Может начаться после Фазы 2. Использует VisitTile из US1.

### Внутри каждой пользовательской истории

- Backend (Rust storage, команды) → Frontend компоненты → Интеграция
- Тесты → реализация (для unit-тестов утилит)

### Возможности для параллелизма

- **T001-T003** — все параллельно (разные файлы)
- **T004-T007** (Rust backend) и **T008-T013** (Frontend) — параллельно
- **T008, T009, T010** — параллельно (разные файлы)
- **T017, T018** (StarRating, ScanUploader) — параллельно
- **T024, T025** (фильтры) — последовательно (T025 зависит от T024)
- **T026, T027, T028** — параллельно (разные модули Rust + утилита)
- **T034** параллельно с T035-T037, но T036/T037 зависят от T034 (VisitCard)
- **US5 зависит от US4** (нужен VisitDetailView из T037 для редактирования/удаления)

## Стратегия реализации

### Сначала MVP (только US1)

1. Завершить Фазу 1: Настройка
2. Завершить Фазу 2: Фундаментальные (КРИТИЧНО — блокирует все US)
3. Завершить Фазу 3: US1 — Ручное добавление
4. **ОСТАНОВИТЬСЯ и ПРОВЕРИТЬ**: Создать запись → увидеть в реестре → открыть карточку
5. Развернуть/продемонстрировать если готово

### Инкрементальная выдача

1. Фаза 1 + 2 → Фундамент готов
2. + US1 → MVP: ручной ввод + реестр
3. + US2 → Фильтрация и сортировка
4. + US3 → LLM распознавание сканов
5. + US4 → Карточка с навигацией
6. + US5 → Редактирование и удаление
7. + Фаза 8 → Полировка, тесты, линтинг

### Стратегия параллельной команды

С несколькими разработчиками:

1. Команда завершает Фазу 1 + 2 вместе
2. После завершения Фундаментальных:
   - Разработчик A: US1 + US5 (модальная форма)
   - Разработчик B: US2 (фильтры, реестр)
   - Разработчик C: US3 (LLM, pdfium)
3. US4 после US1 (нужна VisitTile)

## Примечания

- Задачи [P] = разные файлы, нет зависимостей
- Метка [Story] связывает задачу с конкретной пользовательской историей
- Каждая пользовательская история должна быть независимо завершаема и тестируема
- Фиксировать после каждой задачи или логической группы
- Останавливаться на контрольных точках для независимой проверки истории
- Избегать: неопределённых задач, конфликтов одних и тех же файлов, кросс-исторических зависимостей

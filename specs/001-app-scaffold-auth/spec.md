## Сценарии пользователя и тестирование *(обязательно)*

### Пользовательская история 1 — Первичная регистрация и вход по пин-коду

При первом запуске приложения пользователь видит экран регистрации, где вводит фамилию, имя, дату рождения, пол и задаёт пин-код. После регистрации при каждом последующем запуске приложение запрашивает пин-код для входа.

**Почему этот приоритет**: Без регистрации и входа невозможно использование ни одной функции приложения. Это фундаментальный сценарий, блокирующий все остальные.

**Независимое тестирование**: Можно полностью протестировать, запустив приложение впервые, пройдя регистрацию, закрыв приложение, открыв снова и войдя с помощью пин-кода. Результат — пользователь попадает на основной экран.

**Сценарии приёма**:

1. **Дано** приложение запускается впервые, **Когда** пользователь заполняет все поля регистрации и задаёт пин-код, **Тогда** данные сохраняются, и пользователь попадает на основной экран
2. **Дано** пользователь уже зарегистрирован, **Когда** приложение запускается, **Тогда** отображается экран ввода пин-кода
3. **Дано** экран ввода пин-кода, **Когда** пользователь вводит верный пин-код, **Тогда** открывается основной экран
4. **Дано** экран ввода пин-кода, **Когда** пользователь вводит неверный пин-код, **Тогда** отображается сообщение об ошибке, поле очищается

---

### Пользовательская история 2 — Основной экран и навигация по меню

После успешного входа пользователь видит основной экран с меню навигации и базовой структурой для будущих разделов приложения.

**Почему этот приоритет**: Основной экран — точка входа во все будущие функции (трекинг здоровья, анализы, приёмы). Без него приложение не имеет функционального назначения.

**Независимое тестирование**: Можно протестировать, войдя в приложение и убедившись, что основной экран отображается корректно, меню открывается, пункты меню реагируют на нажатия.

**Сценарии приёма**:

1. **Дано** пользователь вошёл по пин-коду, **Когда** отображается основной экран, **Тогда** видно основное содержимое и доступное меню
2. **Дано** основной экран, **Когда** пользователь открывает меню, **Тогда** отображаются пункты: «Профиль», «Выход»

---

### Пользовательская история 3 — Профиль: просмотр, редактирование, смена пин-кода

Пользователь может просматривать и редактировать свои регистрационные данные, а также менять пин-код.

**Почему этот приоритет**: Пользователю необходимо актуализировать личные данные и обеспечивать безопасность через смену пин-кода. Это вторая по важности функция после входа.

**Независимое тестирование**: Можно протестировать, открыв профиль, изменив фамилию/имя/дату/пол, сохранив, закрыв и открыв приложение снова — изменения должны отобразиться. Аналогично для смены пин-кода.

**Сценарии приёма**:

1. **Дано** пользователь на основном экране, **Когда** открывает профиль, **Тогда** отображаются текущие данные (фамилия, имя, дата рождения, пол)
2. **Дано** экран профиля, **Когда** пользователь редактирует данные и сохраняет, **Тогда** изменения сохраняются и отображаются при повторном открытии
3. **Дано** экран профиля, **Когда** пользователь инициирует смену пин-кода, вводит текущий и новый пин-код, **Тогда** пин-код обновляется, следующий вход — по новому пин-коду
4. **Дано** смена пин-кода, **Когда** пользователь вводит неверный текущий пин-код, **Тогда** операция отклоняется с сообщением об ошибке

---

### Пользовательская история 4 — Выход из приложения

Пользователь может выйти из приложения через меню, после чего при следующем открытии потребуется ввод пин-кода.

**Почему этот приоритет**: Завершение сеанса — базовая функция безопасности. Низкий приоритет, так как не блокирует основную работу.

**Независимое тестирование**: Войти, нажать «Выход», закрыть приложение, открыть снова — должен появиться экран ввода пин-кода.

**Сценарии приёма**:

1. **Дано** пользователь на основном экране, **Когда** выбирает «Выход» в меню, **Тогда** сеанс завершается, приложение закрывается или переходит на экран ввода пин-кода

---

### Граничные случаи

- **Пустые поля при регистрации**: если пользователь пытается зарегистрироваться, не заполнив обязательные поля (фамилия, имя, дата рождения, пол, пин-код), система отображает сообщение об ошибке и не сохраняет данные
- **Слишком короткий или длинный пин-код**: пин-код должен содержать от 4 цифры; при выходе за пределы — сообщение об ошибке
- **Повторный пин-код при регистрации**: пин-код вводится дважды для подтверждения; при несовпадении — сообщение об ошибке
- **Повреждённый файл данных профиля**: если файл профиля повреждён или отсутствует у зарегистрированного пользователя, приложение предлагает повторную регистрацию или восстановление из резервной копии
- **Дата рождения в будущем**: дата рождения не может быть позже текущей даты
- **Множественные неудачные попытки входа**: ограничений на число попыток ввода пин-кода нет — приложение локальное, угроза подбора перебором минимальна

## Экранные формы

### Форма 1: Экран регистрации

**Статус**: НОВАЯ
**Связанная история**: Пользовательская история 1

**Элементы управления**:

| Элемент | Тип | Обязательный | Описание |
|---------|-----|:-----------:|----------|
| Фамилия | Текстовое поле | Да | Фамилия пользователя |
| Имя | Текстовое поле | Да | Имя пользователя |
| Дата рождения | Выбор даты | Да | Дата рождения пользователя |
| Пол | Выпадающий список | Да | Мужской / Женский |
| Пин-код | Поле ввода пароля | Да | Пин-код из цифр (4 символа) |
| Подтверждение пин-кода | Поле ввода пароля | Да | Повторный ввод для подтверждения |
| Зарегистрироваться | Кнопка | Да | Сохраняет данные и переходит на основной экран |

**Логика работы**:

При открытии формы все поля пустые. При нажатии «Зарегистрироваться» выполняется валидация: все поля заполнены, пин-код 4 цифры, пин-код и подтверждение совпадают, дата рождения не в будущем. При ошибках — подсветка невалидных полей с сообщениями. При успехе — данные сохраняются, переход на основной экран. Форма показывается только при первом запуске (когда профиль ещё не создан).

**Статус**: НОВАЯ
**Связанная история**: Пользовательская история 3

**Элементы управления**:

| Элемент | Тип | Обязательный | Описание |
|---------|-----|:-----------:|----------|
| Текущий пин-код | Поле ввода пароля | Да | Для подтверждения личности |
| Новый пин-код | Поле ввода пароля | Да | Новый пин-код (4 цифры) |
| Подтверждение нового пин-кода | Поле ввода пароля | Да | Повторный ввод нового пин-кода |
| Сохранить | Кнопка | Да | Применяет новый пин-код |
| Отменить | Кнопка | Да | Возврат в профиль без изменений |

**Логика работы**:

Пользователь вводит текущий пин-код, новый и подтверждение. При нажатии «Сохранить»: проверяется текущий пин-код, новый пин-код 4 цифры, новый и подтверждение совпадают. При успехе — пин-код обновляется, возврат в профиль. При ошибках — сообщения с подсветкой полей.

## Требования *(обязательно)*

### Функциональные требования

- **FR-001**: Система ДОЛЖНА позволять пользователю зарегистрироваться при первом запуске, введя фамилию, имя, дату рождения и пол
- **FR-002**: Система ДОЛЖНА требовать задания пин-кода (от 4 цифры) при регистрации с подтверждением
- **FR-003**: Система ДОЛЖНА запрашивать пин-код при каждом запуске приложения после завершения регистрации
- **FR-004**: Система ДОЛЖНА отображать основной экран после успешного входа
- **FR-005**: Система ДОЛЖНА предоставлять меню с пунктами «Профиль» и «Выход»
- **FR-006**: Пользователь ДОЛЖЕН иметь возможность просматривать и редактировать свои данные (фамилия, имя, дата рождения, пол)
- **FR-007**: Пользователь ДОЛЖЕН иметь возможность сменить пин-код, подтвердив текущий
- **FR-008**: Система ДОЛЖНА валидировать все обязательные поля: не пустые, дата не в будущем, пин-код 4 цифры, подтверждение пин-кода совпадает
- **FR-009**: Система ДОЛЖНА сохранять данные профиля и пин-код локально в файловой системе
- **FR-010**: Система ДОЛЖНА завершать сеанс при выборе «Выход», требуя повторного ввода пин-кода при следующем запуске
- **FR-011**: Система ДОЛЖНА предоставлять Makefile с целями для установки зависимостей, сборки и запуска приложения в режиме разработки
- **FR-012**: Система ДОЛЖНА отображать понятные сообщения об ошибках при невалидном вводе

### Ключевые сущности

- **Профиль пользователя**: личные данные пользователя — фамилия, имя, дата рождения, пол; хранится в JSON-файле
- **Учётные данные (пин-код)**: хэш пин-кода, привязанный к профилю; хранится отдельно или в том же файле, но в защищённой форме (не в открытом виде)

## Допущения

- Приложение однопользовательское — регистрируется только один человек; нет ролей, мультиаккаунтов, администраторов
- Пин-код — единственный метод аутентификации; нет восстановления через email или контрольные вопросы; при утере пин-кода пользователь вручную удаляет файлы данных приложения из файловой системы и проходит регистрацию заново
- Данные профиля хранятся в одном JSON-файле; пин-код хэшируется перед сохранением (не хранится в открытом виде)
- Makefile используется на всех платформах (Windows через `nmake` или `mingw32-make`, Linux/macOS через стандартный `make`); альтернативно может использоваться `just` или npm-скрипты, если make недоступен на Windows
- Приложение не синхронизирует данные с облаком; всё хранится локально
- Валюты, языки, часовые пояса — язык интерфейса русский по умолчанию; поддержка других языков не входит в объём данной функции

**Входные данные**: Проектная документация из `/specs/002-health-data-entry/`

**Тесты**: Да — спецификация включает тестирование пользовательских сценариев (unit + E2E).

- **[P]**: Можно запускать параллельно (разные файлы, нет зависимостей)
- **[Story]**: Какой пользовательской истории принадлежит задача (например, US1, US2, US3)
- Включайте точные пути к файлам в описаниях

## Зависимости и порядок выполнения

### Зависимости фаз

- **Настройка (Фаза 1)**: Нет зависимостей — можно начинать немедленно
- **Фундаментальные (Фаза 2)**: БЛОКИРУЕТ все пользовательские истории
- **Пользовательские истории (Фаза 3+)**: US1 → US2 → US3 → US4 (последовательно)
- **Полировка (Фаза 7)**: Зависит от завершения всех пользовательских историй

### Зависимости пользовательских историй

- **US1 (P1)**: Может начаться после Фундаментальной (Фаза 2)
- **US2 (P2)**: Зависит от US1 (нужна существующая таблица с данными)
- **US3 (P3)**: Зависит от US1 (нужны замеры для отображения на графиках)
- **US4 (P4)**: Зависит от US1 и US3 (новый показатель должен появиться и в таблице, и на графиках)

### Внутри каждой пользовательской истории

- Backend (IPC-команды) перед Frontend (сервисы и компоненты)
- Сервисы перед компонентами
- Компоненты перед интеграцией

### Возможности для параллелизма

- T001, T002 (Настройка) — параллельно
- T003–T008 (Фундаментальные) — все параллельно (разные файлы)
- T009, T010, T011 (US1 backend) — параллельно
- T013, T014, T015, T016 (US1 утилиты/сервисы/компоненты) — параллельно (разные файлы)
- T020, T021 (US2 backend) — параллельно
- T026, T027 (US3 компоненты) — параллельно
- T035, T036, T037 (Полировка тесты) — параллельно

## Стратегия реализации

### Сначала MVP (только Пользовательская история 1)

1. Завершить Фазу 1: Настройка
2. Завершить Фазу 2: Фундаментальные (КРИТИЧНО — блокирует все истории)
3. Завершить Фазу 3: Пользовательская история 1
4. **ОСТАНОВИТЬСЯ и ПРОВЕРИТЬ**: Таблица → новый замер → заполнение → сохранение → отклонения
5. Развернуть/продемонстрировать если готово

### Инкрементальная выдача

1. Настройка + Фундаментальные → Фундамент готов
2. US1 (таблица + ввод + отклонения + валидация) → Проверить независимо → Демо (MVP!)
3. US2 (редактирование + удаление + подтверждение) → Проверить независимо → Демо
4. US3 (графики + ползунки периода) → Проверить независимо → Демо
5. US4 (расширение показателей) → Проверить → Демо
6. Полировка → Финальное тестирование

## Примечания

- Задачи [P] = разные файлы, нет зависимостей
- Метка [Story] связывает задачу с конкретной пользовательской историей для отслеживания
- Каждая пользовательская история должна быть независимо завершаема и тестируема
- Фиксировать после каждой задачи или логической группы
- Останавливаться на любой контрольной точке для независимой проверки истории
- Избегать: неопределенных задач, конфликтов одних и тех же файлов, кросс-исторических зависимостей, нарушающих независимость

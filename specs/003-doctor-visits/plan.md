# План реализации: Ведение информации о приёмах врача

**Ветка**: `003-doctor-visits` | **Дата**: 2026-04-09 | **Спецификация**: [spec.md](./spec.md)

## Сводка

Реализация полноценного CRUD для записей о приёмах врача с плиточным реестром, модальными формами создания/редактирования, карточкой приёма с навигацией между приёмами одного врача, фильтрацией по периоду и специальности, а также LLM-распознаванием сканов медицинских заключений через конфигурируемый OpenAI-совместимый API. Данные хранятся в JSON-файле, сканы — в отдельной директории файловой системы.

## Технический контекст

**Язык/Версия**: TypeScript 5.8 (frontend), Rust 1.x (backend Tauri)
**Основные зависимости**: React 19, Zod 4, Tauri v2, tailwindcss v4, shadcn/ui, lucide-react, pdfium-render (Rust, для конвертации PDF → PNG)
**Хранилище**: JSON-файлы в AppData (аналогично существующим `health.json`, `profile.json`). Сканы — файлы в директории `scans/` внутри AppData с бизнес-именами (`2026_04_09_Кардиолог_a1b2c3d4.pdf`). Новая сущность: `doctor-visits.json`
**Тестирование**: Vitest (frontend unit-тесты), Rust inline-тесты (backend). В проекте используются ТОЛЬКО модульные (unit) тесты.
**Целевая платформа**: Windows, Linux, macOS (Tauri desktop)
**Тип проекта**: Десктопное приложение (однопользовательское, локальное)
**Цели производительности**: Мгновенная реакция UI при CRUD-операциях (<100мс). LLM-распознавание — асинхронная операция с индикатором (длительность зависит от провайдера).
**Ограничения**:
- Полностью оффлайн-режим для чтения/ручного ввода (NFR-003)
- LLM-распознавание требует доступность OpenAI-совместимого API (локального или удалённого)
- Файлы сканов вне публичного доступа (NFR-001)
- Данные изолированы по текущему пользователю (NFR-002)
**Масштаб/Объем**: Однопользовательское приложение. Типичный объём — десятки/сотни записей о приёмах.

### Неопределённости (NEEDS CLARIFICATION)

| # | Вопрос | Контекст |
|---|--------|----------|
| NC-001 | Формат хранения сканов: копировать ли загруженный файл в AppData или сохранять путь к оригиналу? | Spec говорит «сохранять скан в локальном файловом хранилище» — нужно решить, копируем ли мы файл или храним ссылку |
| NC-002 | Формат IPC-команд для LLM: синхронный или асинхронный вызов? | Распознавание скана может занимать несколько секунд — нужно ли блокировать UI или использовать асинхронный паттерн |
| NC-003 | Поддержка PDF как формата сканов | В допущениях указан PDF как основной формат, но Tauri fs работает с файлами напрямую — PDF требует конвертации в изображение для отображения в UI |
| NC-004 | Структура данных: отдельный JSON-файл или расширение существующего? | Текущие данные (health.json, profile.json) хранятся раздельно — логично создать отдельный файл |
| NC-005 | Формат промпта для LLM | Какая системная подсказка даёт наилучшие результаты для распознавания медицинских документов |

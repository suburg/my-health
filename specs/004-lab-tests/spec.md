# Спецификация функции: Модуль «Анализы»

**Ветка функции**: `004-lab-tests`
**Создана**: 2026-04-10
**Статус**: Черновик
**Входные данные**: Описание пользователя: Новый модуль – Анализы. Пользователь вводит (загружает) информацию о сданных анализах (крови, мочи и прочее). Инструментальные исследования не входят, они вводятся в Приёмы. Состав информации: дата, лаборатория, анализ чего, скан результата, результаты в структурированном виде: показатель, единица измерения, референсное значение (интервал или конкретное значение), фактическое значение, примечание. Ввод должен быть как вручную, так и путем распознавания скана через LLM. Интерфейс аналогичен приёмам – реестр, карточка, форма добавления/изменения. Важно учесть, что в разных лабораториях один и тот же показатель может называться немного иначе. Нужно приводить названия показателей к эталонному виду (для обеспечения сравнимости). Рассмотри различные варианты, откуда можно взять эталонный справочник показателей.

## Сценарии пользователя и тестирование *(обязательно)*

### Пользовательская история 1 — Просмотр реестра анализов

Пользователь открывает модуль «Анализы» и видит список всех сданных анализов, отсортированный по дате (новые сверху). Каждая запись отображается в виде карточки-плитки с ключевой информацией: дата, тип анализа (крови, мочи и т.д.), лаборатория, количество показателей, индикатор наличия скана. Пользователь может фильтровать записи по периоду дат, типу анализа и лаборатории.

**Почему этот приоритет**: Базовый сценарий — пользователь должен видеть все свои анализы в одном месте. Без реестра остальные функции бесполезны. Это MVP модуля.

**Независимое тестирование**: Можно полностью протестировать, добавив несколько записей анализов через seed-данные и проверив их отображение, сортировку и фильтрацию в реестре.

**Сценарии приёма**:

1. **Дано** в системе есть 5 записей анализов за разные даты, **Когда** пользователь открывает модуль «Анализы», **Тогда** отображаются все 5 записей, отсортированные по убыванию даты
2. **Дано** пользователь открыл реестр, **Когда** применяет фильтр по периоду «с 01.01.2026 по 31.03.2026», **Тогда** отображаются только записи в этом диапазоне
3. **Дано** в реестре есть записи с разными типами анализов, **Когда** пользователь выбирает фильтр «Кровь», **Тогда** показываются только анализы крови

---

### Пользовательская история 2 — Ручной ввод анализа с показателями

Пользователь создаёт новую запись анализа: указывает дату, лабораторию, тип анализа (крови, мочи и т.д.), прикрепляет скан (опционально) и вручную вводит показатели — каждый показатель включает название, фактическое значение, единицу измерения, референсное значение (интервал или конкретное значение) и опциональное примечание. Система автоматически подсвечивает отклонения от референсных значений.

**Почему этот приоритет**: Основной способ наполнения модуля данными. Без ручного ввода пользователь не может ввести информацию, если скан недоступен или LLM не используется.

**Независимое тестирование**: Можно полностью протестировать, создав анализ вручную с 3-5 показателями, проверив валидацию, сохранение и корректное отображение в карточке.

**Сценарии приёма**:

1. **Дано** пользователь открыл форму создания анализа, **Когда** заполняет все обязательные поля (дата, лаборатория, тип анализа, минимум 1 показатель) и нажимает «Сохранить», **Тогда** записи появляется в реестре
2. **Дано** пользователь ввёл показатель с фактическим значением, выходящим за референсный интервал, **Когда** форма отображает этот показатель, **Тогда** значение подсвечивается как отклонение (визуальный индикатор)
3. **Дано** пользователь не указал дату, **Когда** нажимает «Сохранить», **Тогда** отображается ошибка валидации, сохранение не происходит
4. **Дано** пользователь ввёл несколько показателей, **Когда** сохраняет анализ, **Тогда** все показатели сохраняются и отображаются в карточке

---

### Пользовательская история 3 — Распознавание скана анализа через LLM

Пользователь загружает скан (фотографию или PDF) бланка лаборатории. Система отправляет скан в LLM для распознавания и автоматически заполняет все доступные данные: дату анализа, лабораторию, тип материала (кровь, моча и т.д.), описание анализа, а также все показатели с их значениями, единицами измерения и референсными интервалами. Пользователь проверяет и корректирует распознанные данные перед сохранением. Система приводит распознанные названия показателей к эталонному виду с помощью справочника.

**Почему этот приоритет**: Значительно ускоряет ввод — пользователю не нужно вручную набирать 20-30 показателей из бумажного бланка. Критичная функция для удобства, но не блокирующая MVP (ручной ввод уже работает).

**Независимое тестирование**: Можно протестировать, загрузив тестовый скан с известными данными, и проверив, что LLM корректно распознал ≥80% показателей и система привела названия к эталонному виду.

**Сценарии приёма**:

1. **Дано** пользователь загрузил скан бланка анализа, **Когда** нажимает «Распознать», **Тогда** система возвращает распознанные данные для предзаполнения всех полей формы: дата, лаборатория, тип анализа, описание и показатели
2. **Дано** LLM распознал дату, лабораторию и тип анализа из скана, **Когда** данные отображаются в форме, **Тогда** соответствующие поля (дата, лаборатория, тип анализа) предзаполнены и доступны для редактирования
3. **Дано** LLM распознал показатели и метаданные, **Когда** данные отображаются в форме, **Тогда** пользователь может отредактировать любое поле — как общие данные, так и каждый показатель
4. **Дано** скан не удалось распознать (ошибка LLM, плохое качество), **Когда** происходит сбой, **Тогда** пользователю отображается понятное сообщение об ошибке и предлагается ручной ввод

### Пользовательская история 4 — Эталонные названия показателей при вводе и распознавании

При создании анализа пользователь выбирает название показателя из справочника — все предлагаемые названия уже являются эталонными. При распознавании через LLM справочник передаётся модели как контекст, и LLM сама приводит названия к эталонному виду. Пользователь видит эталонное название; оригинальное (распознанное) сохраняется для справки, если оно отличается. Это обеспечивает сопоставимость показателей across разных лабораторий и дат.

**Почему этот приоритет**: Ключевое требование — без нормализации невозможно сравнивать показатели между анализами из разных лабораторий, что снижает ценность модуля. Однако модуль может работать и без полной нормализации (показатели сохраняются как есть), поэтому это не блокирует MVP.

**Независимое тестирование**: Можно протестировать, создав два анализа из разных лабораторий с разными названиями одного показателя, и проверив, что система привела их к одному эталонному названию.

**Сценарии приёма**:

1. **Дано** пользователь вводит название показателя вручную, **Когда** начинает печатать, **Тогда** автодополнение предлагает эталонные названия из справочника
2. **Дано** пользователь выбрал показатель из справочника, **Когда** строка заполняется, **Тогда** сохраняется эталонное название показателя
3. **Дано** LLM распознала показатель с названием, отличающимся от эталонного, **Когда** данные отображаются в форме, **Тогда** показывается эталонное название, а оригинальное (распознанное) сохраняется для справки

---

### Пользовательская история 5 — Просмотр детальной карточки анализа с историей показателей

Пользователь кликает на записи в реестре и переходит на детальную страницу анализа. Карточка содержит: общую информацию (дата, лаборатория, тип анализа), скан (изображение/PDF), таблицу показателей с цветовой индикацией отклонений, прикреплённые файлы. Для каждого показателя отображается его предыдущее значение — значение того же показателя из самого последнего анализа того же типа с указанием даты и ссылкой на тот анализ. Это позволяет отслеживать динамику изменений показателей. Доступна навигация к предыдущему/следующему анализу того же типа, а также редактирование и удаление.

**Почему этот приоритет**: Завершающий пользовательский опыт — детальное просмотр и управление. Зависит от историй 1 и 2, но ценен сам по себе для демонстрации.

**Независимое тестирование**: Можно протестировать, открыв детальную карточку существующего анализа, проверив отображение всех блоков, навигацию и функции редактирования/удаления.

**Сценарии приёма**:

1. **Дано** пользователь кликнул на записи в реестре, **Когда** открывается детальная страница, **Тогда** отображаются все данные анализа: общая информация, таблица показателей, скан
2. **Дано** у анализа есть показатели с отклонениями, **Когда** отображается таблица, **Тогда** отклонения подсвечены (например, красным — выше нормы, синим — ниже)
3. **Дано** пользователь нажимает «Редактировать», **Когда** открывается форма, **Тогда** все данные предзаполнены и доступны для изменения
4. **Дано** для показателя существует предыдущее измерение в другом анализе того же типа, **Когда** отображается таблица показателей, **Тогда** рядом с текущим значением показывается предыдущее значение с датой и ссылкой на тот анализ
5. **Дано** у показателя нет предыдущего измерения (это первый анализ такого типа), **Когда** отображается таблица, **Тогда** поле предыдущего значения пустое или отображается «—»

### Граничные случаи

- **Пустой реестр**: Пользователь впервые открыл модуль — отображается пустое состояние с подсказкой «Добавьте первый анализ»
- **Очень большое количество показателей**: Бланк содержит 50+ показателей — таблица показателей поддерживает прокрутку и группировку по категориям
- **Некорректный скан**: Пользователь загрузил фото, не являющееся бланком анализа — LLM возвращает пустой результат или ошибку, пользователь вводит вручную
- **Дублирование показателей**: LLM распознал один показатель дважды — пользователь видит дубликаты и может удалить лишний
- **Референсный интервал не указан**: Лаборатория не указала референсные значения на бланке — поле референса остаётся пустым, отклонение не определяется
- **Нечисловой показатель**: Показатель с текстовым значением (например, «Цвет мочи: янтарный») — не сравнивается числово, а проверяется на вхождение в список допустимых значений. Если список не задан — подсветка не применяется
- **Ручная привязка не найдена**: Пользователь не смог найти подходящий эталонный показатель в диалоге привязки — диалог закрывается, показатель остаётся с пометкой «не распознан»
- **Удаление анализа**: При удалении анализов пользователь должен подтвердить действие (диалог подтверждения)

Сохранение через сервис с атомарной записью.

### Таблица показателей (детальная логика)

Таблица показателей — динамическая сетка редактирования, позволяющая добавлять, изменять и удалять строки. Каждая строка соответствует одному показателю.

**Структура строки:**

| Колонка | Тип ввода | Описание |
|---------|-----------|----------|
| Название показателя | Текстовое поле с автодополнением | При вводе предлагает эталонные названия из встроенного справочника показателей. При выборе — автоматически подставляются эталонное название, единица измерения и тип значения. Пользователь также может ввести произвольное название — оно сохранится без эталонного соответствия |
| Фактическое значение | Зависит от типа | Для числовых показателей — числовое поле. Для нечисловых (например, «Цвет мочи») — текстовое поле или выпадающий список допустимых значений, если они определены в справочнике |
| Единица измерения | Текстовое поле | Единица измерения, указанная на бланке (предзаполняется из справочника при выборе показателя, но может быть изменена). Для нечисловых показателей — пустое или «—» |
| Референс | Контекстный ввод | Для числовых показателей: поля «от» и «до» (интервал) ИЛИ поле «конкретное значение». Для нечисловых: список допустимых значений (например, для «Цвет мочи»: «соломенно-жёлтый, жёлтый, янтарный») |
| Примечание | Текстовое поле | Опциональное примечание к показателю |
| Кнопка удаления | Кнопка | Удаляет строку из таблицы |

**Поведение при добавлении строки:**

- Нажатие «Добавить показатель» создаёт пустую строку с фокусом на поле «Название показателя»
- Если пользователь начинает вводить название — срабатывает автодополнение по справочнику (по эталонному названию и известным синонимам). Все предлагаемые варианты — уже эталонные названия
- При выборе показателя из справочника автоматически заполняются: эталонное название, единица измерения, тип значения (числовой/текстовый). Для числовых показателей — пустые поля референса (от/до/значение), для нечисловых — пустое поле допустимых значений
- Если пользователь вводит произвольное название (не выбирая из справочника) — оно сохраняется как есть, без эталонного соответствия; тип значения по умолчанию — числовой

**Поведение при распознавании через LLM:**

- При вызове LLM система передаёт скан и справочник показателей (эталонные названия и известные синонимы) как контекст нормализации
- LLM распознаёт все данные из скана (дату, лабораторию, тип анализа, описание, показатели) и **сразу приводит названия показателей к эталонному виду**, используя переданный справочник
- Если LLM нашла соответствие в справочнике — возвращает эталонное название и распознанное оригинальное (как было на бланке)
- Если LLM не смогла подобрать соответствие — возвращает показатель с распознанным названием и пометкой «не распознан»
- Для каждого показателя предзаполняются: эталонное название, оригинальное название, тип значения (берётся из справочника по эталонному названию), фактическое значение, единица измерения, референсные поля
- Если LLM не распознала референсные значения (на бланке не указаны или нечитаемо) — поля референса/допустимых значений остаются пустыми
- Если LLM не удалось распознать тип значения по эталонному названию (показатель новый), тип определяется по характеру распознанного значения (число → числовой, иначе текстовый)

**Поведение при вводе фактического значения:**

- При каждом изменении поля «Фактическое значение» выполняется мгновенное сравнение с референсом:
  - **Числовой показатель, референс задан интервалом (от/до):** значение в диапазоне → зелёный; значение > до → красный; значение < от → синий
  - **Числовой показатель, референс задан конкретным значением:** совпадает → зелёный; не совпадает → красный
  - **Числовой показатель, референс не задан:** без подсветки (серый)
  - **Нечисловой показатель, допустимые значения заданы:** фактическое значение есть в списке допустимых → зелёный; отсутствует → красный
  - **Нечисловой показатель, допустимые значения не заданы:** без подсветки (серый)
- Подсветка применяется к ячейке фактического значения и/или к фону строки

**Поведение при удалении строки:**

- Удаление строки происходит мгновенно без дополнительного подтверждения (аналогично редактированию в форме)
- Если в таблице остаётся 0 строк — кнопка «Сохранить» блокируется (минимум 1 показатель обязателен)

**Валидация строки:**

- Название показателя — обязательно (минимум 1 символ)
- Фактическое значение — обязательно (число для числовых показателей, непустая строка для нечисловых)
- Единица измерения — обязательна для числовых показателей (минимум 1 символ), для нечисловых — опциональна
- Референс/допустимые значения — опциональны: могут быть не указаны, но если указаны — для числовых показателей должен быть либо полный интервал (от и до), либо конкретное значение; для нечисловых — непустой список допустимых значений

## Требования *(обязательно)*

### Функциональные требования

- **FR-001**: Система ДОЛЖНА предоставлять реестр всех записей анализов с сортировкой по дате (новые сверху)
- **FR-002**: Система ДОЛЖНА позволять фильтровать анализы по периоду дат и типу анализа
- **FR-003**: Система ДОЛЖНА позволять создавать новые записи анализов через модальную форму
- **FR-004**: Система ДОЛЖНА позволять редактировать и удалять существующие записи анализов
- **FR-005**: Каждая запись анализа ДОЛЖНА содержать: дату, лабораторию, тип анализа, опциональное описание («анализ чего»), опциональный скан, список показателей
- **FR-006**: Каждый показатель ДОЛЖЕН содержать: название (эталонное), оригинальное название (если отличается), тип значения (числовой/текстовый), фактическое значение (число или текст), единицу измерения (для числовых), референсное значение (интервал min/max или конкретное значение — для числовых; список допустимых значений — для текстовых), опциональное примечание
- **FR-007**: Система ДОЛЖНА визуально подсвечивать отклонения фактических значений от референсных (выше нормы, ниже нормы, в норме)
- **FR-008**: Система ДОЛЖНА поддерживать загрузку сканов бланков анализов (изображения: JPG, PNG, WebP; документы: PDF)
- **FR-009**: Система ДОЛЖНА распознавать из сканов через LLM и предзаполнять форму: дату анализа, лабораторию, тип материала (анализ чего), описание, а также все показатели
- **FR-010**: Пользователь ДОЛЖЕН иметь возможность проверить и отредактировать все распознанные LLM данные — как общие поля (дата, лаборатория, тип анализа), так и каждый показатель — перед сохранением
- **FR-011**: Система ДОЛЖНА приводить названия показателей к эталонному виду с использованием комбинированного справочника: при распознавании через LLM справочник передаётся как контекст — LLM приводит названия к эталонному виду; при ручном вводе — автодополнение предлагает эталонные названия из справочника; для редких показателей без соответствия — возможность ручной привязки пользователем
- **FR-012**: Если при распознавании через LLM название показателя не найдено в справочнике, система сохраняет показатель с распознанным LLM названием и помечает как «не распознан» (без эталонного соответствия). При ручном вводе произвольное название сохраняется без пометки
- **FR-019**: Система ДОЛЖНА предоставлять возможность ручной привязки показателя с пометкой «не распознан» к эталонному названию из справочника через диалог выбора с поиском по справочнику. Полная форма управления справочником (создание новых показателей, редактирование синонимов) не входит в текущий объём и может быть вынесена в отдельную задачу
- **FR-013**: Система ДОЛЖНА хранить сканы в отдельной директории файловой системы с именованием по бизнес-правилам (аналогично приёмам: `YYYY_MM_DD_тип_анализа_uuid.ext`)
- **FR-014**: Система ДОЛЖНА поддерживать навигацию «предыдущий» / «следующий» анализ в рамках одного типа анализа
- **FR-015**: Система ДОЛЖНА поддерживать автодополнение названия лаборатории на основе ранее введённых значений
- **FR-016**: Данные анализов ДОЛЖНЫ храниться в JSON-файле с полем `schemaVersion` для версионирования структуры
- **FR-017**: При удалении анализа система ДОЛЖНА требовать подтверждения пользователя
- **FR-018**: Система ДОЛЖНА корректно обрабатывать отсутствие, повреждение или несовместимую версию файла данных анализов

### Ключевые сущности

- **LabTest (Анализ)**: Представляет одну сдачу анализа. Ключевые атрибуты: ID, дата, лаборатория, тип анализа (кровь/моча/биохимия/гормоны/прочее), описание («анализ чего»), путь к скану, список показателей, createdAt, updatedAt.
- **LabTestIndicator (Показатель анализа)**: Конкретный измеренный показатель. Ключевые атрибуты: эталонное название показателя, оригинальное название (если распознано иначе), тип значения (числовой/текстовый), фактическое значение (число или строка), единица измерения (для числовых), референсный минимум (опционально, для числовых), референсный максимум (опционально, для числовых), референсное конкретное значение (опционально, для числовых), допустимые значения (список строк, опционально, для нечисловых), примечание.
- **LabTestIndicatorReference (Справочник показателей)**: Справочная запись эталонного показателя. Ключевые атрибуты: эталонное название, известные синонимы/вариации названий, тип значения (числовой/текстовый), тип анализа (кровь/моча/...), единица измерения (для числовых), тип референса (интервал/значение/список), типовые референсные интервалы (для числовых), допустимые значения (для текстовых, например: «соломенно-жёлтый, жёлтый, янтарный» для цвета мочи). Справочник используется для нормализации названий при вводе и распознавании.

### Источники эталонного справочника показателей

Для нормализации названий показателей необходим справочник, который сопоставляет различные варианты названий (из разных лабораторий) с единым эталонным названием. Рассмотрены подходы:

| Подход | Описание | Применимость |
|--------|----------|-------------|
| Встроенный справочник основных показателей | Набор наиболее распространённых показателей (общий анализ крови, мочи, биохимия) поставляется с приложением | Надёжная основа, работает без интернета; используется при ручном вводе (автодополнение) и передаётся LLM при распознавании |
| Нормализация через LLM | Справочник передаётся LLM как контекст — LLM сама приводит названия к эталонному виду при распознавании | Гибкость для редких показателей, автоматическое определение типа анализа по материалу, но зависит от качества модели и размера контекста |
| Открытые медицинские классификаторы | Использование международных стандартов (LOINC, SNOMED CT) и российских справочников | Авторитетный источник, но могут быть лицензионные ограничения |
| Пользовательская привязка | Возможность вручную связать нераспознанное название с эталонным показателем | Позволяет расширять справочник в процессе использования |

Рекомендуется комбинированный подход: встроенная база основных показателей + LLM для дополнительных (с автоматическим определением типа анализа по материалу) + ручная привязка новых.

## Допущения

- **А1**: Приложение работает в одно-пользовательском режиме — все анализы принадлежат одному пользователю (соответствует конституции)
- **А2**: Все данные хранятся локально в JSON-файлах, без сетевой синхронизации (соответствует конституции)
- **А3**: LLM для распознавания сканов настраивается через существующий конфигурационный файл (аналогично модулю «Приёмы») — используются те же настройки LLM API. LLM распознаёт не только показатели, но и метаданные: дату, лабораторию, тип анализа, описание
- **А4**: Справочник показателей хранится как встроенный JSON-файл в приложении с возможностью расширения пользователем (добавление новых соответствий). При распознавании через LLM справочник передаётся в системном промпте целиком как контекст для нормализации названий. Ожидаемый размер справочника — до ~200 записей
- **А5**: Типы анализов (кровь, моча, биохимия, гормоны, генетика, прочее) — фиксированный перечень, определяемый на уровне конфигурации
- **А6**: Единицы измерения не нормализуются — сохраняются так, как указаны лабораторией (разные лаборатории могут использовать разные единицы для одного показателя)
- **А7**: Референсные значения могут различаться у разных лабораторий для одного показателя — сохраняются те, что указаны в бланке конкретной лаборатории
- **А8**: Визуальная подсветка отклонений использует стандартные цветовые индикаторы (красный — выше нормы / не в списке допустимых, синий — ниже нормы, зелёный — в норме) в соответствии со светлой темой приложения
- **А9**: Большинство показателей — числовые, поэтому справочник по умолчанию содержит числовые показатели. Нечисловые показатели (цвет, прозрачность, реакция и т.п.) добавляются в справочник по мере необходимости

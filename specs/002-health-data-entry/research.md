# Research: Ввод и визуализация данных о здоровье

**Дата**: 2026-04-08
**Ветка**: `002-health-data-entry`

## Решение 1: Библиотека графиков

**Decision**: Recharts

**Rationale**: Recharts — нативная React-библиотека, работает напрямую с JSX-компонентами, что идеально вписывается в стек React 19 + TypeScript. Поддерживает:
- Несколько осей Y через `<YAxis yAxisId="...">` — критично для отображения показателей с разными единицами
- `Brush` компонент — горизонтальный ползунок для фильтрации по временному диапазону
- TypeScript-типизация из коробки
- Компонентная композиция — графики строятся декларативно через JSX

Chart.js требует обёрток и работы с Canvas, что усложняет интеграцию с React и кастомизацию осей.

**Alternatives considered**:
- **Chart.js** — популярен, но требует `react-chartjs-2`, работает через Canvas, сложнее с multi-axis
- **D3.js** — максимальная гибкость, но значительно сложнее в интеграции, требует низкоуровневой работы с DOM

## Решение 2: Структура хранения health-данных

**Decision**: Два JSON-файла — `health.json` (замеры) и `metric-config.json` (определения показателей)

**health.json** — массив замеров, каждый замер — объект с датой как ключом и значениями показателей:
```json
{
  "schemaVersion": 1,
  "entries": [
    {
      "date": "2026-04-08",
      "metrics": {
        "height": { "value": 175 },
        "weight": { "value": 73 },
        "bloodPressure": { "systolic": 120, "diastolic": 80 },
        "sleep": { "minutes": 450 }
      }
    }
  ]
}
```

**Rationale**: Один файл на все замеры — простое чтение/запись, атомарная замена всего файла. Структура `metrics` как объект позволяет легко добавлять новые показатели без изменения схемы. Разные типы значений (простое число, пара, длительность) унифицированы через объектную обёртку.

**metric-config.json** — массив определений показателей:
```json
{
  "schemaVersion": 1,
  "metrics": [
    {
      "key": "weight",
      "label": "Вес",
      "unit": "кг",
      "type": "number",
      "range": { "min": 30, "max": 300 },
      "autofill": false,
      "order": 2,
      "category": "anthropometry"
    },
    {
      "key": "bloodPressure",
      "label": "Давление",
      "unit": "мм рт.ст.",
      "type": "compound",
      "compoundFields": ["systolic", "diastolic"],
      "compoundLabels": ["Верхнее", "Нижнее"],
      "range": { "systolic": { "min": 70, "max": 250 }, "diastolic": { "min": 40, "max": 150 } },
      "autofill": false,
      "order": 4,
      "category": "cardio"
    },
    {
      "key": "sleep",
      "label": "Сон",
      "unit": "ч:м",
      "type": "duration",
      "range": { "min": 60, "max": 900 },
      "autofill": false,
      "order": 6,
      "category": "sleep"
    }
  ]
}
```

**Rationale**: Конфигурация отдельно от данных — позволяет разработчику добавлять показатели без изменения кода. Поле `order` определяет порядок строк в таблице. `category` — для разделительных линий между блоками.

**Alternatives considered**:
- **Один файл с инлайн-конфигом** — неудобно для расширения, смешивает данные и метаданные
- **Хранение каждого замера в отдельном файле** — усложняет чтение всей таблицы, атомарность сложнее

## Решение 3: Inline-редактирование таблицы

**Decision**: Кастомный компонент EditableCell на основе `<input>` с обработкой `onBlur` / `onKeyDown(Enter)`

**Rationale**: Не требует внешних библиотек editable-таблиц (например, AG Grid, TanStack Table), которые избыточны для нашего сценария. Простая HTML-таблица (`<table>`) с кастомными ячейками даёт полный контроль над стилизацией и поведением. Фиксация первого столбца — CSS `position: sticky; left: 0`.

**Alternatives considered**:
- **TanStack Table** — мощная, но добавляет сложность и зависимости для простого случая
- **ContentEditable div** — проблемы с фокусом и валидацией, сложнее контролировать ввод

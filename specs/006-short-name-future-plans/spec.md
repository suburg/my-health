# Спецификация функции: Модуль «Планы»

**Ветка функции**: `006-future-plans`
**Создана**: 2026-04-13
**Статус**: Черновик
**Входные данные**: Описание пользователя: Планы — Реестр плановых задач на будущее: плановые визиты к врачу, исследования, анализы. Реестр, карточка, форма создания/редактирования. Состав: вид (приём, анализ, исследование), плановая дата, обязательное (да/нет), описание, статус (Запланировано/Выполнено/Отменено). Действия: Выполнить (фактическая дата), Отменить (причина).

## Сценарии пользователя и тестирование *(обязательно)*

### Пользовательская история 1 — Просмотр реестра плановых задач

Пользователь открывает модуль «Планы» и видит список всех плановых задач, отсортированный по плановой дате (ближайшие сверху). Каждая задача отображается в виде карточки-плитки с ключевой информацией: вид задачи, плановая дата, статус, признак обязательности. Пользователь может фильтровать записи по виду задачи и по статусу.

**Почему этот приоритет**: Базовый сценарий — пользователь должен видеть все запланированные задачи в одном месте. Без реестра остальные функции бесполезны. Это MVP модуля.

**Независимое тестирование**: Можно полностью протестировать, добавив seed-данные и проверив отображение, сортировку и фильтрацию в реестре.

**Сценарии приёма**:

1. **Дано** в системе есть 5 плановых задач с разными датами, **Когда** пользователь открывает модуль «Планы», **Тогда** отображаются все 5 задач, отсортированных по возрастанию плановой даты
2. **Дано** пользователь применил фильтр по виду «Приём», **Когда** реестр обновляется, **Тогда** отображаются только задачи типа «Приём»
3. **Дано** пользователь применил фильтр по статусу «Запланировано», **Когда** реестр обновляется, **Тогда** отображаются только запланированные задачи
4. **Дано** пользователь открыл модуль впервые без задач, **Тогда** отображается пустое состояние с подсказкой «Добавьте первую задачу»

---

### Пользовательская история 2 — Создание плановой задачи

Пользователь создаёт новую плановую задачу: указывает вид (приём, анализ, исследование), плановую дату, признак обязательности, описание. Задача создаётся со статусом «Запланировано».

**Почему этот приоритет**: Основной способ наполнения модуля данными. Без создания задач учёт планов невозможен.

**Независимое тестирование**: Можно полностью протестировать, создав задачу вручную, проверив валидацию обязательных полей, сохранение и отображение в реестре.

**Сценарии приёма**:

1. **Дано** пользователь открыл форму создания задачи, **Когда** заполняет обязательные поля (вид, плановая дата) и нажимает «Сохранить», **Тогда** задача появляется в реестре со статусом «Запланировано»
2. **Дано** пользователь не указал вид задачи, **Когда** нажимает «Сохранить», **Тогда** отображается ошибка валидации
3. **Дано** пользователь не указал плановую дату, **Когда** нажимает «Сохранить», **Тогда** отображается ошибка валидации

---

### Пользовательская история 3 — Выполнение и отмена плановой задачи

Пользователь отмечает запланированную задачу как «Выполнено» (указывает фактическую дату выполнения) или «Отменено» (опционально указывает причину отмены). Задача меняет статус, сохраняется в реестре с визуальным отличиением по статусу.

**Почему этот приоритет**: Ключевой рабочий процесс модуля — управление жизненным циклом плановой задачи от «Запланировано» до «Выполнено» или «Отменено».

**Независимое тестирование**: Можно протестировать, выбрав существующую запланированную задачу и выполнив действия «Выполнить» и «Отменить», проверив смену статуса и сохранение данных.

**Сценарии приёма**:

1. **Дано** задача имеет статус «Запланировано», **Когда** пользователь нажимает «Выполнить» и указывает фактическую дату, **Тогда** статус меняется на «Выполнено», сохраняется фактическая дата
2. **Дано** задача имеет статус «Запланировано», **Когда** пользователь нажимает «Отменить» без указания причины, **Тогда** статус меняется на «Отменено», причина отсутствует
3. **Дано** пользователь нажимает «Отменить» и указывает причину, **Тогда** статус меняется на «Отменено», причина сохраняется

---

### Пользовательская история 4 — Просмотр детальной карточки плановой задачи

Пользователь кликает на задачу в реестре и переходит на детальную страницу. Карточка содержит: вид, плановую дату, признак обязательности, описание, статус, историю изменений (фактическая дата или причина отмены). Доступны редактирование и удаление.

**Почему этот приоритет**: Завершающий пользовательский опыт — детальный просмотр и полное управление задачей. Зависит от историй 1–3, но ценен для навигации и редактирования.

**Независимое тестирование**: Можно протестировать, открыв детальную карточку существующей задачи, проверив отображение всех полей и функции редактирования/удаления.

**Сценарии приёма**:

1. **Дано** пользователь кликнул на задачу в реестре, **Когда** открывается детальная страница, **Тогда** отображаются все данные задачи
2. **Дано** пользователь нажимает «Редактировать», **Когда** открывается форма, **Тогда** все данные предзаполнены и доступны для изменения
3. **Дано** пользователь нажимает «Удалить», **Когда** подтверждает действие, **Тогда** задача удаляется из реестра
4. **Дано** задача имеет статус «Запланировано», **Когда** пользователь открывает детальную страницу и нажимает «Выполнить», **Тогда** открывается окно выбора фактической даты, после подтверждения статус меняется на «Выполнено»
5. **Дано** задача имеет статус «Запланировано», **Когда** пользователь нажимает «Отменить», **Тогда** открывается окно ввода причины отмены, после подтверждения статус меняется на «Отменено»

---

### Граничные случаи

- **Пустой реестр**: Пользователь впервые открыл модуль — отображается пустое состояние с подсказкой «Добавьте первую задачу»
- **Плановая дата в прошлом при создании**: Система разрешает создать задачу с плановой датой в прошлом, но сразу после сохранения задача визуально помечается как «просроченная» (оранжевый индикатор)
- **Плановая дата в прошлом (существующая задача)**: Задача со статусом «Запланировано» и прошедшей плановой датой визуально выделяется (просроченная задача)
- **Выполнение задачи с датой в будущем**: Фактическая дата может быть любой — в прошлом или будущем
- **Отмена уже выполненной задачи**: Система не позволяет отменить задачу со статусом «Выполнено» — действие недоступно
- **Удаление задачи**: При удалении пользователь должен подтвердить действие (модальное окно подтверждения)
- **Редактирование завершённой задачи**: Задачу со статусом «Выполнено» или «Отменено» можно редактировать — система предупреждает, что изменение статуса вернёт задачу в «Запланировано»

### Функциональные требования

- **FR-001**: Система ДОЛЖНА предоставлять реестр всех плановых задач с сортировкой по плановой дате (ближайшие сверху)
- **FR-002**: Система ДОЛЖНА позволять фильтровать задачи по виду (приём, анализ, исследование), по статусу (запланировано, выполнено, отменено) и по признаку обязательности. Все фильтры комбинируются (логика AND)
- **FR-003**: Система ДОЛЖНА позволять создавать новые плановые задачи через модальную форму
- **FR-004**: Каждая плановая задача ДОЛЖНА содержать: вид (приём/анализ/исследование), плановую дату (ДД.ММ.ГГГГ), признак обязательности (да/нет), описание (до 500 символов, опционально), статус (запланировано/выполнено/отменено)
- **FR-005**: Система ДОЛЖНА позволять отметить запланированную задачу как «Выполнено» с указанием фактической даты выполнения
- **FR-006**: Система ДОЛЖНА позволять отметить запланированную задачу как «Отменено» с опциональной причиной отмены (до 300 символов)
- **FR-007**: Система ДОЛЖНА позволять редактировать существующие плановые задачи через модальную форму
- **FR-008**: Система ДОЛЖНА позволять удалять плановые задачи с подтверждением пользователя
- **FR-009**: Система ДОЛЖНА визуально отображать статус и просроченность задачи цветовыми индикаторами: «Запланировано» — синий, «Выполнено» — зелёный, «Отменено» — серый, «Просрочено» (плановая дата в прошлом, статус «Запланировано») — оранжевый

- **FR-011**: При редактировании задачи со статусом «Выполнено» или «Отменено» система ДОЛЖНА предупреждать, что статус будет сброшен в «Запланировано»

### Ключевые сущности

Ключевые атрибуты: ID, вид (приём/анализ/исследование), плановая дата, обязательная (да/нет), описание, статус (запланировано/выполнено/отменено), фактическая дата (для выполненных), причина отмены (для отменённых), createdAt, updatedAt.

## Допущения

- **А1**: Приложение работает в одно-пользовательском режиме — все плановые задачи принадлежат одному пользователю (соответствует конституции)
- **А2**: Все данные хранятся локально в JSON-файлах, без сетевой синхронизации (соответствует конституции)
- **А3**: Интерфейс модуля следует паттерну модулей «Приёмы», «Анализы» и «Препараты» — реестр с плитками, модальная форма, детальная карточка
- **А4**: Фильтры в реестре применяются по кнопке «Применить»
- **А5**: Просроченные задачи — это задачи со статусом «Запланировано», у которых плановая дата меньше сегодняшней
- **А6**: При редактировании завершённой задачи (выполнена/отменена) статус автоматически сбрасывается в «Запланировано», фактическая дата и причина отмены очищаются
- **А7**: Визуальная индикация статусов: «Запланировано» — синий, «Выполнено» — зелёный, «Отменено» — серый, «Просрочено» — оранжевый (в соответствии со светлой темой приложения)
- **А8**: Фактическая дата выполнения по умолчанию предлагается как сегодня при открытии окна выполнения
